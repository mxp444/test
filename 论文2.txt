摘  要
互联网平台在金融产品推广与信息扩散中扮演的角色日益突出，与之伴生的是虚假理财宣传、非法集资、高回报诱导、数字货币骗局及私域引流等违规信息的泛滥。这些风险内容呈现出跨平台蔓延、图文交织传播、伪装手段多样化的态势，给传统人工巡查和简单关键词匹配带来了严峻挑战——后者往往在面对复杂舆情时显得力不从心。基于这一现实困境，本文着手设计并落地了一套依托多模态信息处理技术的网络金融舆情风险监测系统。
系统整体遵循 B/S 架构思路搭建。后端选用 Python 语言配合 Flask 框架，承担接口服务、权限校验、采集调度与分析调度等职责；前端则借助 HTML、CSS 和 JavaScript 完成界面呈现与用户交互；数据层面依托 MongoDB 存放来自多个平台的舆情原始数据及其分析产出。在数据来源上，系统覆盖微博、抖音、百度贴吧、小红书等主流社交媒体，并将各平台数据归整为字段相对统一的文档结构。对于采集到的图文混合舆情，系统分别从帖子正文、配图视觉内容、OCR 提取文字三条路径抽取风险特征——涵盖金融属性判定、风险要素识别、情感极性、煽动程度、二维码存在性、图片清晰度、色彩饱和度及版式设计感等维度，最终经由多模态融合机制输出综合风险评分、风险等级划分、风险成因说明与处置建议。
功能层面，系统按角色做了明确切分。面向普通用户的前台业务端提供监测首页、舆情浏览、风险分析结果、单条详情报告以及个人追踪等功能入口；面向管理员的后台管理端则涵盖平台维护、采集任务调控、模型状态查看、运行日志审阅、账号与用户管理、系统参数配置及数据库运维等操作。经过开发与测试验证，系统能够串联起多平台舆情采集、风险研判、结果可视化与后台管控的完整链路，为网络金融领域的舆情风险监测工作提供了一种切实可行的软件实现路径。
关键词:网络金融舆情；多模态信息处理；风险监测；MongoDB；Flask

Abstract
With the increasing role of Internet platforms in financial information dissemination and product marketing, risk-related information such as false financial promotion, illegal fundraising, high-return inducement, cryptocurrency scams, and private-domain traffic diversion has shown the characteristics of multi-platform diffusion, mixed text-image communication, and strong concealment. Traditional methods based on manual inspection or simple keyword matching are insufficient for identifying complex public opinion risks in a timely manner. To address these problems, this thesis designs and implements an online financial public opinion risk monitoring system based on multimodal information processing.
The system adopts a B/S architecture. The backend is implemented with Python and Flask, providing interface services, permission control, crawling task scheduling, and analysis task scheduling. The frontend is developed with HTML, CSS, and JavaScript for page display and user interaction. MongoDB is used to store multi-platform public opinion data and analysis results. The system supports the collection of financial-related public opinion data from Weibo, Douyin, Baidu Tieba, Xiaohongshu and other platforms, and converts data from different platforms into a relatively unified document structure. For the collected text-image public opinion data, the system extracts risk features from post text, image content, and OCR text, including financial attributes, risk elements, sentiment tendency, incitement degree, QR codes, image clarity, color richness, and design features. Based on multimodal fusion, the system generates comprehensive risk scores, risk levels, risk reasons, and handling suggestions.
In terms of system functions, the system distinguishes between ordinary users and administrators. Ordinary users can view the monitoring homepage, public opinion monitoring results, risk analysis results, detailed reports, and personal tracking records on the frontend. Administrators can manage platforms, crawling tasks, model status, operation logs, accounts, users, system settings, and database data through the backend. The implementation and testing results show that the system can complete multi-platform public opinion collection, risk analysis, result presentation, and backend management, providing a feasible software system solution for online financial public opinion risk monitoring.
Keywords：online financial public opinion; multimodal information processing; risk monitoring; MongoDB; Flask



第1章  绪  论
1.1  课题背景及研究的目的与意义
眼下，互联网早已成为金融产品推广与营销的核心阵地。微信朋友圈里的理财广告、抖音快手上自称"投资导师"的短视频博主、各类论坛中流传的所谓"内幕消息"——金融信息几乎渗透到了网络空间的每一个角落。但问题在于，这些信息中夹杂着数量可观的风险因素：虚假宣传、非法集资、庞氏骗局，不一而足。相较于传统线下集资活动，网络金融舆情的传播速度更快、波及面更广、隐蔽性也更强。一条经过精心包装的投资广告，短短几小时便可触达数万受众，而监管部门却很难在第一时间察觉并介入处置。更棘手的是，这些风险信息的载体形式多样——文字帖子、宣传图片、短视频兼而有之，仅凭关键词过滤远不足以实现全面识别。
从技术视角审视，近些年多模态信息处理领域的进展为解决上述难题提供了新的可能。自然语言处理（NLP）、图像识别、视频分析等技术的综合运用，使得对网络舆情的立体化监测成为现实。举例来说，NLP 技术可以捕捉文字中的高收益承诺表述，OCR 技术能够从图片里提取联系方式，视频分析则有助于检测营销话术的使用。本课题的出发点正是构建这样一套多模态网络金融舆情监测系统——它能够自动从社交媒体、论坛等平台抓取数据，从中甄别金融风险信息并发出预警。这项工作对于防范网络金融风险、守护公众财产安全而言，具有切实的应用价值。

1.2.1  国内研究现状
多模态数据指的是对同一数据对象在不同空间维度下的描述形式[1]。从发文量来看，2012年至2019年间国内多模态领域在知网上的相关论文数量偏少，2020年之后则出现了大幅度的持续攀升，反映出该领域在国内学术界受到的关注度显著提高[2]。就研究方法而言，深度学习是近年来国内多模态研究的主要技术路线，研究方向集中在图像融合、图像分类、人脸识别、多任务学习、医学影像等方面，横跨人工智能、医学、社交网络等多个应用领域。
在人工智能领域，特征提取、图像融合和视觉问答构成了多模态技术的几个应用热点。黄学坚[3]等人设计了一种融合外部知识与证据的场景图注意力网络，将其应用于多模态谣言检测任务，该网络能够有效捕捉图文之间的细粒度语义关联，在公开数据集上取得的准确率与F1值均明显超过基线模型。他[4]另外还给出了一种基于样本内外协同表示和自适应融合的多模态学习方案。清华大学的王鑫[5]围绕多模态数据与人工智能的结合开展了系统性研究，探讨了跨媒体关联分析与语义理解的相关技术路径，以及多模态数据统一表达与融合的方法论。
在医学领域，多模态数据的内涵更为具体——它涵盖诊断报告以及X光、CT、磁共振成像、超声、正电子发射断层扫描等多种医疗成像设备产生的影像资料[6]。胡馨悦[7]等人将DenseNet与Mamba架构相结合，提出了多模态脑影像模型DenseMamba，在ADNI数据集上对阿尔茨海默症实现了精准分类，准确率达到92.42%，表现优于已有方法。
在日常社交场景中，多模态技术同样得到了广泛应用。王楠[8]等研究者针对模态缺失导致模型性能大幅下滑这一难题，设计了注意力不确定模态蒸馏框架（AUMDF）。该框架借助动态权重调整和掩码Transformer应对不确定的模态输入，并创新性地引入知识蒸馏技术，在多个基准数据集上有效改善了模型的鲁棒性与准确率。张晋敏等[9]则提出了一种基于BiGRU模型的多模态网络舆情情感分析框架，该框架将文本特征与图像特征通过线性融合技术加以整合，以期获得更为准确的情感判断结果。
纵观国内多模态研究，可以归纳出应用驱动、技术纵深化的鲜明走向。其长处体现在两个方面：
其一，前沿技术与垂直行业的结合程度较深。
其二，研究者对模态缺失、数据不确定性等实际部署中的瓶颈问题给予了充分重视，并提出了多种针对性方案来增强模型鲁棒性。
不过，也存在若干值得正视的短板：
其一，底层架构层面的原始创新仍显不足。多数工作依然是在Transformer、GRU等现有主干网络基础上做改进，真正从零开始的模型架构探索和预训练范式革新相对稀缺。
其二，技术组合的巧妙程度高于理论层面的根本性突破。现有研究更多是将已有技术进行精巧整合以解决特定问题，而在多模态认知机理、统一表征理论等基础性课题上，尚缺乏足够多从0到1的开创性成果。
笔者还注意到一个现实问题：国内多模态模型在金融舆情风险监测——甚至仅仅是一般性舆情监测——中的应用仍然相当有限。当前金融舆情风险的监测手段主要停留在文本挖掘与关键词匹配层面，虽然能够捕获部分显性风险信号，但局限性十分明显。一方面，对于那些包装在专业图表、收益截图或宣传海报中的隐性风险——比如通过OCR才能提取到的夸大收益承诺、嵌入图片中的非法联系方式——传统方法基本无能为力。另一方面，单一文本分析缺乏对多模态信息进行交叉验证的能力：当帖子文字描述"稳健收益"而配图却展示陡峭的高回报曲线时，纯文本分析无法洞察这种图文之间的矛盾。再者，面对短视频、直播等动态内容中的违规营销行为，传统人力巡查手段同样缺乏有效的监测途径，导致对新型网络金融风险的识别存在明显的滞后和遗漏。
1.2.2  国外研究现状
Ehsan Elhamifar[10]提出了一种融合RGB、光流和音频的多模态元学习框架，在少量样本条件下实现了视频动作的精准时序定位。俞启凡[11]给出了DataTailor方案——一个聚焦于信息量、独特性和代表性的协作式多模态数据选择框架，其目标是缓解视觉指令数据激增所带来的冗余和计算开销问题。Wu K[12]提出了SkySense++，这是一个专门面向地球观测任务的多模态遥感基础模型，在覆盖农业、林业、灾害管理等7大领域的12项任务上，其分类、检测和分割性能均超越了此前的最优模型，展现出突出的泛化能力。潘张[13]等人发布了开源多模态奖励模型InternLM-XComposer2.5-Reward (IXC-2.5-Reward)，旨在应对大型视觉语言模型输出不可靠且缺乏公开奖励模型的困境。Ranjan Sapkota教授[14]等人系统梳理了多模态大语言模型在农业领域的应用现状、面临的挑战及未来前景。梁宇轩[15]等人则聚焦"多模态时间序列分析"这一方向，明确了数据异构、模态鸿沟等核心难点，在此基础上提出了"融合、对齐、迁移"的统一跨模态交互框架来归类现有方法，同时汇总了相关数据集与模型资源，并对实际应用和未来走向做了展望。
从上述工作可以看出，国外多模态大模型研究正在从单纯的性能追赶转向更深层次的"智能重构"阶段，核心关切集中在可靠推理、高效训练与安全鲁棒性三个维度。一方面，研究者通过架构创新与训练机制改进追求更强的模型能力；另一方面，学界开始系统性地直面多模态融合本身带来的新挑战——Ehsan Elhamifar的元学习框架致力于攻克少样本时序定位难题，Muchao Ye等人则深入揭示了多模态特性引入的独特安全隐患（攻击面扩大、模态未对齐等）。与此同时，农业、医疗、地球科学等垂直领域的落地探索（如Ranjan Sapkota团队的工作），以及梁宇轩等人对多模态时间序列分析的框架性梳理，共同表明该领域正朝着深入化、专业化的实际应用方向演进，并在此过程中正面应对数据异构、模态鸿沟等根本性挑战。整体而言，该领域在拓展性能边界的同时，对模型内在机制的稳健性、安全性与领域适配能力给予了越来越多的关注。
1.3 主要研究内容
本毕业设计的核心目标在于开发一套立足多模态信息处理的网络金融舆情风险监测系统，具体工作涵盖以下几个方面：
一，多源舆情数据的自动化采集。系统需要具备从微博、论坛、贴吧等3至5个主流平台自动抓取金融相关舆情数据的能力，数据形式包括文字、图片、视频等。采集过程应体现一定的智能性，能够依据关键词、标签等条件筛选出与金融话题相关的内容。
二，文本风险识别模型的构建。面向文字类舆情，开发基于NLP的风险识别模型，使其能够捕捉虚假宣传特征（如"保本保息""日赚千元"）、非法集资迹象（如"拉人头""返利"）、诈骗话术痕迹（如"内幕消息""限时优惠"）等风险要素。
三，图片内容的检测与识别。面向图片类舆情，运用OCR技术提取图片中的文字信息并送入风险识别流程。同时，系统还应具备识别图片中金融相关元素的能力——二维码、微信号、银行卡号等均属于需要关注的对象。
四，综合风险评估体系的建立。将文本、图片等不同模态的分析结果加以融合，对每条舆情给出综合风险评分。评估所考量的维度包括金融属性强度、风险关键词命中数量、煽动性程度、传播范围等。
五，可视化监测平台的开发。将上述功能整合封装为Web平台，提供舆情监测、风险预警、数据统计等业务入口。平台应能实时呈现监测结果，并对高危舆情做出标注和提醒。

最后为结论部分。对本文完成的主要工作加以总结，说明系统实现和测试所达到的效果，同时分析当前工作的局限——数据源覆盖范围有限、模型训练样本规模不足、图像分析能力仍有提升空间、多模态融合规则较为依赖人工权重等，并就后续扩展更多舆情平台、优化风险识别模型、增强视频内容分析能力、完善预警机制和提升系统实用性等方向做出展望。
第2章  系统分析
2.1 系统建设背景分析
互联网平台正逐步演变为金融信息传播和产品营销的关键渠道，网络金融舆情的辐射范围与影响深度也随之持续扩大。在微博、抖音、百度贴吧、小红书等平台上，正常的金融知识分享和投资经验交流与虚假理财宣传、非法集资、高收益诱导、数字货币骗局以及借助二维码和联系方式进行私域引流的风险内容并存。这类信息惯用"稳赚不赔""限时名额""内幕消息"等话术吸引用户注意力，容易对普通网民的投资决策产生误导，严重时甚至引发财产损失乃至群体性风险事件。
与传统舆情相比较，网络金融舆情在几个方面表现出显著差异：传播速度快、覆盖面广、隐蔽性强、表现形式复杂。部分风险信息并不直接以文字形式呈现，而是将联系方式、收益截图、营销海报、二维码等关键内容藏匿于图片之中，单纯依靠人工巡查或关键词检索很难做到及时发现。面对多平台、多类型、图文混合的舆情数据，有必要建设一个兼具多平台采集、智能检索、风险分析和详情报告展示能力的系统，将分散的舆情内容转化为可查询、可分析、可追踪的结构化信息，进而提升风险发现的效率。
正是基于上述考量，本系统面向金融监管人员、平台管理人员和舆情监测人员进行建设。系统前台主要承担监测结果展示、风险分析结果查阅等任务，后台则负责平台配置、采集任务管理、运行日志查看、账号管理等工作。前台展示与后台管理各司其职，既能满足日常监测的使用需求，又便于管理员对采集和运行过程实施统一维护。
2.2 系统业务分析流程分析
2.2.1 舆情数据采集流程
舆情数据采集构成本系统业务流程的起始环节，主要由管理员在后台采集任务管理模块中操作完成。管理员依据实际监测需要，先选定需要采集的平台——微博、抖音、百度贴吧、小红书等数据来源均在可选范围内——随后对采集规则加以配置，配置项主要包括采集关键词、cookie等。系统接收到管理员提交的采集任务后，按照所选平台分别启动对应的采集流程。
由于各平台的数据组织形式和访问入口存在差异，系统在采集过程中采取差异化策略。获取页面或接口返回内容后，系统对帖子正文、作者信息等字段进行解析。随后执行关键词过滤和去重判断，将无关内容或重复数据拦截在数据库之外。
采集完成后，各平台的数据分别写入MongoDB中对应的集合。需要强调的是，采集得到的原始舆情数据并不直接作为最终风险结论，而是继续流入文本分析、图像检测等后续处理环节。普通用户主要在前台查看经过采集和分析后的舆情结果，采集任务的控制权则归属于管理员后台功能。

2.2.2 舆情风险分析流程
舆情风险分析流程是系统在数据采集完成后的核心处理环节，其任务是将MongoDB中保存的原始舆情数据进一步转化为可供展示的风险分析结果。微博、抖音、百度贴吧、小红书等平台的采集数据写入数据库后，系统会筛选出尚未完成分析且同时包含文本和图片的舆情记录作为分析对象，自动或手动触发多模态分析，并将分析结果写回MongoDB，再由前端读取展示。
具体分析过程中，系统先读取舆情正文和本地图片路径，对正文内容执行文本风险识别。正文分析完成后，系统继续对图片内容进行检测——判断图片是否带有营销引流、宣传包装或视觉诱导特征。与此同时，系统通过OCR提取图片中的文字，并将OCR文本再次送入文本风险分析流程，以便识别隐藏在宣传海报、收益截图或联系方式图片中的风险信息。
在获得正文文本、图片特征和OCR文本三类分析结果之后，系统进入多模态融合阶段。该阶段综合不同模态中的金融属性、风险关键词、煽动性、二维码、联系方式和视觉宣传特征，计算综合风险分，并据此划分为低风险、中风险、中高风险或高风险等级别。分析结果最终写回MongoDB，前端依据这些结果进行详情展示。

图2-2  舆情风险分析业务流程图
2.2.3 风险预警与报告查看流程
风险预警与报告查看流程属于多模态风险分析完成后的结果展示环节。系统将舆情正文、图片、OCR文本和融合分析结果写回MongoDB后，在前端进行呈现。普通用户登录系统后，可在舆情监测页面浏览风险结果。
当用户发现某条舆情需要进一步关注时，可点击对应条目进入详情报告页面。系统根据舆情标识读取单条数据，在报告页面展示帖子原文、图片、综合风险分、风险等级等内容。借助详情报告，用户不仅能看到风险等级本身，还能了解系统做出该判断的具体依据，便于后续研判和处理。
对于需要持续关注的重点舆情，用户可将其加入个人追踪列表。系统按用户维度保存追踪记录，方便用户后续集中查阅相关舆情。普通用户的操作范围主要集中在查看采集后的舆情内容和分析结果，复杂配置和运行过程则由后台管理员负责维护。

2.2.4 管理员与普通用户操作流程
本系统依据用户角色划分出普通用户操作流程和管理员操作流程，以确保前台业务使用与后台系统维护之间保持清晰边界。普通用户登录后进入前台业务端，其操作围绕舆情结果查看和个人业务展开：进入舆情监测页面了解系统采集到的舆情概况和风险分布，通过平台、关键词、风险等级、分析状态和图片条件等方式进行高级检索，查看不同风险等级的舆情内容，点击具体舆情进入详情报告页面查看原文、图片、风险原因和分析建议。普通用户还可以提交新的采集源申请，将重点舆情加入个人追踪列表，以及维护个人资料和登录密码。
管理员登录后进入后台管理端，承担系统运行和资源维护方面的职责。管理员可进行平台管理，处理普通用户提交的采集源申请，并根据实际需要配置平台访问参数、关键词、是否要求图片等采集规则。在采集任务管理中，管理员可选择平台并启动、暂停或继续采集任务，也可对历史未分析数据执行补分析。管理员还可查看运行日志和模型状态，进行账号管理、用户管理和系统设置——包括维护账号角色、用户操作范围、系统运行参数以及重置分析引擎等。
经由上述角色划分，系统形成了"前台看结果、后台管运行"的操作格局。普通用户面向监测结果、风险分析、详情报告和个人追踪等业务功能，管理员则负责采集配置、模型状态、运行日志、账号权限和系统参数维护。这种划分契合前后台分离的设计思路，能够避免普通用户接触复杂配置和运行过程，使系统功能结构更加清晰。

2.3 系统可行性分析
从技术可行性角度来看，本系统选用Flask、MongoDB、原生HTML/CSS/JavaScript、多平台爬虫和多模态分析模块进行开发，整体技术路线比较成熟，契合本科毕业设计中中小型Web系统的实现需求。Flask本身结构轻量，便于搭建后端接口和组织业务逻辑；原生前端技术足以支撑系统页面展示、筛选交互和报告查看等基本需求；MongoDB的文档型存储方式天然适合保存微博、抖音、百度贴吧、小红书等不同平台来源的半结构化舆情数据。
从经济可行性角度来看，本系统的开发主要依赖开源技术和本地运行环境，Python、Flask、MongoDB、HTML、CSS、JavaScript等核心技术均不涉及高额许可费用。系统无需依赖大型商业软件平台，普通开发环境即可完成主要功能的开发、调试和运行。就毕业设计而言，该方案能够以较低成本完成多平台采集、舆情存储、风险分析和前端展示等功能，经济投入可控，具备良好的实现条件。
从操作可行性角度来看，系统按照普通用户和管理员两类角色划分操作流程，有效降低了不同用户的使用复杂度。普通用户主要在前台业务端查看监测首页、舆情监测结果、风险分析结果和详情报告，并可进行采集源申请、个人追踪和个人资料维护；管理员则在后台管理端负责平台管理、采集任务控制、运行日志查看、模型状态查看、账号管理、用户管理和系统设置等工作。这种角色边界的划定有利于保持系统操作逻辑的清晰性。
从应用可行性角度来看，网络金融舆情传播快、形式复杂、风险隐蔽，单纯依靠人工巡查难以及时发现高风险内容。本系统面向金融监管人员、平台管理人员和舆情监测人员，能够支撑多平台金融风险舆情的采集、检索、分析和报告查看。
综合以上分析，本系统在技术、经济、操作和应用层面均具备可行性，能够满足毕业设计阶段的系统建设要求。
2.4功能需求分析
2.4.1 数据源配置与管理需求
网络金融舆情来源分散，微博、抖音、百度贴吧、小红书等平台在内容组织方式、访问入口、检索规则和数据结构上各有不同，系统因而需要具备数据源配置与管理功能。借助该功能，管理员可在后台对不同平台的采集规则进行统一维护——平台启用状态、关键词、是否要求图片以及必要的访问配置等均在维护范围之内——从而保障采集任务的稳定执行。
在业务使用层面，普通用户并不直接维护系统采集源，而是在前台提交新增采集源申请。用户填写平台名称、目标网址、关键词等基本信息后提交，系统将其保存为待审核状态。管理员在后台查看申请详情，可根据实际需要补充访问配置、采集页数、是否要求图片等参数，并决定通过或退回。只有审核通过的采集源才会出现在前台采集源列表中，供普通用户查看和使用。这一设计既能满足多平台采集的灵活配置需求，又能防止普通用户误操作影响系统运行，符合前台业务端与后台管理端分离的要求。
2.4.2 多平台舆情采集需求
多平台舆情采集是本系统获取风险监测数据的基础功能。网络金融风险信息分布在不同互联网平台中，单一数据来源难以全面反映舆情传播态势，系统因此需要支持微博、抖音、百度贴吧和小红书四类平台的数据采集。各平台的数据入口和内容组织方式不尽相同，采集策略也应有所区别。多平台采集的意义在于扩大系统对网络金融舆情的覆盖范围。
在采集内容方面，系统应能获取平台来源、作者昵称、正文内容、发布时间、图片路径等信息，并尽量保持字段结构的统一。采集功能还需支持管理员对关键词、访问配置等规则进行设置，以适应不同平台的采集特点和监测需求。采集过程中还应完成图片下载、关键词过滤、重复数据跳过、异常平台跳过和运行日志记录等处理，保证采集结果的有效性和可维护性。
2.4.3 舆情数据存储与去重需求
舆情数据存储与去重是保障系统后续分析和前端展示准确性的基础需求。系统采集的数据来自微博、抖音、百度贴吧、小红书等不同平台，各平台原始字段存在差异，写入数据库前需要统一为相对一致的数据结构——主要包括平台标识、平台名称、帖子ID、作者昵称、正文内容、发布时间、匹配关键词、话题标签、图片本地路径、采集时间等字段。系统还需保留分析状态、分析结果和实际分析图片路径等字段，为后续多模态风险分析、风险筛选和详情报告展示提供数据基础。
存储方式上，系统选用MongoDB保存舆情数据，并按平台分别写入对应集合，便于实现平台隔离、分类查询和后期维护。由于网络平台内容可能被重复采集，系统需要具备去重能力——对于重复数据，应跳过或进行必要更新，避免同一舆情在数据库和前端页面中重复出现，从而提高数据质量和展示效果。

2.4.4 文本风险识别需求
文本风险识别是系统进行舆情风险分析的重要基础环节。网络金融舆情正文中可能包含虚假理财宣传、高收益诱导、保本承诺、拉人头返利、数字货币骗局、非法外盘交易、贷款征信诈骗、私域引流等风险信息，仅靠人工阅读难以及时发现并归纳风险特征。系统需要对采集到的帖子正文进行自动识别和结构化分析，为后续风险评分和报告展示提供依据。
文本风险识别功能应首先判断舆情内容的金融属性类别，确认其是否与理财、投资、贷款、数字货币等金融场景相关；其次识别文本中的风险要素，统计命中词汇、风险类别和出现数量；同时分析文本情感倾向，评估是否存在夸张宣传、限时诱导、强烈号召等煽动性表达。分析结果应写入MongoDB，并在前端详情报告中展示风险类别、风险原因、风险词汇、情感结果和处置建议等内容。系统还需对OCR提取出的图片文字再次执行文本风险分析，以发现隐藏在宣传图片中的金融风险内容。
2.4.5 图片内容检测需求
图片内容检测是系统进行多模态风险识别的重要组成部分。网络金融风险舆情中，部分关键信息并不直接出现在帖子正文中，而是以宣传海报、收益截图、二维码、联系方式、产品介绍图等形式隐藏在配图中。如果系统只对文本内容进行分析，容易遗漏图片中的高收益承诺、私域引流入口和金融营销话术，对帖子配图进行专门检测因而十分必要。
图片内容检测功能需满足以下需求：第一，系统应能判断图片中是否存在二维码或其他引流入口，用于发现将用户引导至外部渠道的风险线索；第二，需要检测图片的清晰度或模糊度，分析图片色彩丰富程度，并评估图片是否具有较强设计感和营销宣传特征；第三，系统应通过OCR提取图片中的关键文字及位置信息，为后续文本风险识别提供输入。
OCR提取出的图片文字需继续进入文本风险识别流程，用于识别图片中的保本宣传、高收益诱导、联系方式、私域引流等风险内容。图片检测结果应与正文文本分析结果共同写入MongoDB中的分析字段，并在舆情详情报告中展示，为后续多模态融合评分、风险原因说明和处置建议生成提供依据。如此一来，系统不仅关注帖子文字，还能识别图文混合舆情中隐藏的金融风险信息。
2.4.6 多模态融合评分需求
系统应具备多模态融合评分能力，将文本分析结果、图片检测结果和舆情传播数据进行特征融合后综合处理，形成统一的风险分数和风险等级。网络金融舆情中的风险信息往往并非由单一因素决定——有些内容正文风险较低，但图片中包含高收益宣传和联系方式；有些内容文字诱导性较强，但传播范围较小。系统需要综合多个维度进行判断，避免仅依靠单一关键词或单一模型导致误判。
多模态融合评分应以文本风险特征和图像风险特征为基础。文本方面可考虑金融属性强度、风险关键词数量、煽动性表达、虚假承诺和紧迫营销等因素；图片方面可考虑OCR文字中的风险词、二维码、手机号、微信号、金额数字、收益率和宣传海报特征等因素。系统还可结合点赞数、评论数、转发数等互动数据，评估该条舆情的传播影响力。不同维度的分析结果转化为同一维度的特征token进行融合后输入构建好的神经网络，最终得到综合风险分数。
系统应根据风险分数划分风险等级，如高风险、中风险和低风险。高风险舆情应在前端页面中进行醒目标注并进入风险预警列表；中风险舆情需提示用户进一步核查；低风险舆情则作为普通监测数据展示。

2.4.7 风险预警展示需求
风险预警展示是系统将多模态分析结果转化为用户可理解信息的重要环节。系统在完成文本、图片和OCR内容的融合评分后，需要将每条舆情的综合风险分、风险等级、风险原因和处置建议直观展示给用户，帮助舆情监测人员快速发现需要重点关注的内容。网络金融风险舆情传播快、隐蔽性强且图文混合，前端展示不能仅停留在原始帖子列表层面，而应结合风险等级、分析状态和平台来源等信息，对舆情进行分类呈现和重点标注。
在展示功能上，系统应支持用户按照平台来源、关键词、风险等级、情感倾向、分析状态等条件筛选舆情，使用户能够从大量采集数据中快速定位目标内容。监测首页主要展示总体概览、风险等级分布、平台来源分布和最新中高风险或高风险舆情；舆情监测页面主要展示舆情列表，并突出平台、作者、正文摘要、图片、风险分数和分析状态等信息；风险分析页面则重点展示风险分类分布、情感分布以及中风险、中高风险、高风险等不同等级内容，便于用户进行集中研判。
2.4.8 舆情详情报告需求
舆情详情报告是系统对单条舆情进行深入展示的重要功能。舆情列表通常只能呈现平台、作者、正文摘要、风险等级等概括信息，无法完整说明风险来源和分析依据。系统需要为单条舆情提供详情报告页面，使用户能够查看原始内容、分析过程和最终判断结果。
详情报告应展示平台来源、作者昵称、发布时间、正文内容、图片、命中关键词、综合风险分、风险等级、风险原因、分析结论和处置建议等基础信息。报告还需展示文本分析结果——金融属性、风险要素、情感倾向和煽动性结果；展示图片检测结果——二维码检测、模糊度、色彩丰富度、设计感和OCR文字；展示多模态融合结果，说明正文文本、图片特征和OCR文本对综合风险判断的贡献。借助详情报告，普通用户可以更直观地理解风险形成原因，舆情监测人员也可据此进行后续人工复核和风险研判。
2.4.9 用户账号与权限管理需求
用户账号与权限管理是保障系统正常使用和后台安全维护的重要需求。本系统同时面向普通用户和管理员两类角色，不同角色的操作范围存在明显差异。若不进行权限划分，普通用户可能误操作采集任务、平台配置、系统设置或账号管理等后台功能，影响系统运行稳定性。系统需要通过账号与权限管理机制，将前台业务功能和后台管理功能区分开来，使不同用户只能访问与自身角色相匹配的功能。
在账号管理方面，系统需支持管理员对账号进行统一维护——新增账号、修改用户名、显示名称、联系电话、所属部门、密码等基本信息，并根据实际情况启用或禁用账号、重置密码以及设置账号角色。通过账号状态和角色设置，系统可控制用户是否能够登录以及登录后能够访问哪些页面和功能。
在权限管理方面，系统需区分普通用户和管理员。普通用户主要使用前台业务端，可查看舆情监测结果、风险分析页面和舆情详情报告，也可提交采集源申请、加入个人追踪并维护个人资料；管理员则负责平台配置、采集任务启动与暂停、运行日志查看、账号管理、用户管理和系统参数维护等后台功能。系统还需记录用户操作信息——登录、修改资料、重置密码等操作——便于管理员查看和追踪系统使用情况。通过上述设计，可以明确普通用户与管理员的权限边界，提高系统管理的规范性和可维护性。
2.5 系统非功能需求分析
2.5.1 可用性需求
可用性需求主要体现为系统界面清晰、功能入口明确、操作过程易于理解。本系统面向金融监管人员、平台管理人员和舆情监测人员，用户未必具备复杂技术背景，系统需要按照角色和业务场景来组织页面。普通用户主要使用前台业务端，可查看监测首页、数据采集申请、舆情监测、个人追踪、风险分析和个人设置等功能；管理员则主要使用后台管理端，负责平台管理、采集任务管理、模型管理、运行日志、账号管理、用户管理和系统设置等工作。
在具体使用体验上，系统应保证导航结构清晰、页面功能分区明确，避免将普通用户不需要的采集配置、运行日志和系统参数暴露在前台。舆情监测页面应支持平台、关键词、风险等级、分析状态、图片条件等筛选方式，并支持帖子正文展开和收起，便于用户快速查看重点内容。详情报告页面应集中展示风险原因、分析结果和处置建议，使用户能够理解系统判断依据。常用操作应尽量通过按钮、下拉框、复选框等方式完成，减少用户理解成本，提高系统整体可用性。
2.5.2 可扩展性需求
可扩展性需求主要体现在系统能够适应后续平台接入、规则调整和分析能力升级。网络金融舆情来源复杂，风险形式也会随平台生态和传播方式的变化而演变，系统不能只面向单一平台或固定规则来设计。当前系统已支持微博、抖音、百度贴吧和小红书四个平台，并通过统一的爬虫运行器维护平台ID、入口文件和MongoDB集合映射，使不同平台能够按照各自采集策略运行，同时在入库后转换为统一字段结构，为后续扩展新平台奠定基础。
在功能扩展方面，新增平台应能通过前台采集源申请和后台平台管理完成配置，由管理员维护平台规则和采集参数。不同平台可采用不同采集入口和采集策略，但前端应通过统一接口展示舆情数据，避免为每个平台重复开发页面。通过上述设计，系统能够在不大幅改变整体结构的前提下适应后续功能扩展。
2.5.3 可维护性需求
可维护性需求主要体现在系统后续修改、配置调整和运行维护的便利性上。本系统包含多平台爬虫、MongoDB数据存储、多模态风险分析、前端展示和后台管理等多个组成部分，如果模块边界不清晰，后续修改和问题定位将较为困难。系统需要按照前端、后端、爬虫和多模态分析模块进行分目录管理，使不同功能之间保持相对独立。四个平台的爬虫应分别放置，并通过统一运行器进行调度，便于后续单独维护或扩展平台。
在配置维护方面，系统运行参数、平台配置、采集源申请、用户账号、个人追踪等信息应通过JSON配置文件或MongoDB进行集中保存，避免分散管理造成维护困难。后台应提供平台管理、采集任务管理、运行日志、模型状态、账号管理和系统设置等功能，方便管理员统一维护系统运行状态。系统还需记录采集日志和用户操作记录，以便在出现采集失败、分析异常或权限操作问题时进行排查，提高系统后续维护效率。
2.6 本章小结
本章围绕网络金融舆情风险监测系统的建设需求展开了系统性分析，为后续设计与实现工作奠定基础。从系统建设背景出发，分析了当前网络金融信息在微博、抖音、百度贴吧、小红书等平台快速传播的现实状况——虚假理财宣传、非法集资、高收益诱导、数字货币骗局和私域引流等风险内容传播速度快、隐蔽性强、图文混合，单纯依靠人工巡查和简单关键词检索难以及时发现风险，建设具备多平台采集、智能检索、风险分析和报告展示能力的舆情监测系统确有必要。
本章还对系统主要业务流程进行了分析，包括舆情数据采集流程、舆情风险分析流程、风险预警与报告查看流程以及管理员与普通用户操作流程。系统以多平台舆情采集为数据入口，将微博、抖音、百度贴吧、小红书等平台的图文内容统一存储到MongoDB中，再通过文本分析、图片检测、OCR识别和多模态融合评分生成风险结果。前端根据风险等级、分析状态和综合风险分进行展示，用户可进一步查看舆情详情报告并对重点内容进行个人追踪。
在可行性分析方面，本章从技术、经济、操作和应用四个角度说明了系统建设的可行性。系统选用Flask、MongoDB、原生前端、多平台爬虫和多模态分析模块，技术路线较为成熟，开发成本较低，适合毕业设计阶段的系统实现需求。系统面向金融监管人员、平台管理人员和舆情监测人员，能够满足多平台金融风险舆情监测、检索、分析和报告查看等实际应用需求。
本章还对用户角色、功能需求、非功能需求和数据需求进行了分析。系统明确区分普通用户和管理员两类角色：普通用户主要使用前台业务端，完成监测结果查看、舆情检索、风险分析、详情报告查看、采集源申请和个人追踪等操作；管理员主要使用后台管理端，负责平台配置、采集任务管理、运行日志查看、模型状态查看、账号管理、用户管理和系统参数维护等工作。前台侧重结果展示和简单业务操作，后台侧重系统配置和运行维护，权限边界较为清晰。
综合来看，本章从需求层面对系统的建设背景、业务流程、角色权限、功能范围、非功能要求和数据内容进行了较为全面的分析，明确了系统应实现的核心业务和运行边界，为后续系统总体架构设计、数据库设计、功能模块设计以及系统实现提供了依据。
第3章  系统设计
3.1 系统设计目标与设计原则
3.1.1 系统设计目标
本系统的总体设计目标是建设一个面向金融监管人员、平台管理人员和舆情监测人员的网络金融舆情风险监测系统，使其能够对多平台网络金融舆情进行采集、存储、分析和展示。
在业务功能设计上，系统应实现多平台舆情采集、统一数据存储、多模态风险分析、风险结果展示、舆情详情报告查看、个人追踪和后台管理等功能。采集到的舆情数据经过文本分析、图片检测、OCR识别和多模态融合后，应能形成综合风险分、风险等级、风险原因、分析结论和处置建议，并在前端页面中直观展示，帮助用户快速识别需要重点关注的金融风险内容。
在系统结构设计上，应明确区分前台业务端和后台管理端。普通用户主要使用前台功能，完成舆情查看、风险分析结果浏览、详情报告查看、采集源申请和个人追踪等操作；管理员则负责后台维护——平台配置、采集任务管理、运行日志查看、模型状态查看、账号管理、用户管理和系统参数设置等。通过清晰的权限划分，可以避免普通用户误操作后台配置，保证系统运行过程更加规范。
系统还应具备一定的可扩展性和可维护性。后续在接入新平台、调整采集规则时，应尽量减少对整体结构的影响，从而提高系统后期维护和功能扩展的便利性。
3.1.2 系统设计原则
系统设计过程中，应遵循实用性、模块化、可扩展性、可维护性、安全性和一致性等原则，使系统既能满足网络金融舆情监测的实际需求，又便于后续维护和功能扩展。
系统设计应坚持实用性原则。本系统面向金融监管人员、平台管理人员和舆情监测人员，核心目标并非单纯展示页面，而是围绕舆情采集、风险分析、结果展示和后台管理形成完整业务流程。前台业务端主要面向普通用户，提供监测首页、舆情监测、风险分析、详情报告、个人追踪等功能；后台管理端主要面向管理员，提供平台配置、采集任务、运行日志、模型管理、账号管理、用户管理和系统设置等功能。
系统应遵循模块化设计原则。整体可划分为前端展示模块、Flask后端服务模块、多平台爬虫模块、MongoDB数据存储模块和多模态风险分析模块，各部分功能相对独立，通过接口和统一数据结构进行连接。
系统还应具备可扩展性和可维护性。不同平台的采集规则、关键词、页数、时间范围等参数可通过后台平台管理和配置文件维护；舆情数据按平台分别存入MongoDB集合，但字段结构保持统一，便于后续接入新平台。
3.2 概要设计
3.2.1 系统总体架构设计
本系统采用B/S架构进行设计，用户通过浏览器访问系统页面，前端负责页面展示与交互，后端负责业务处理、数据调度和接口服务。前端主要使用HTML、CSS、JavaScript实现，后端采用Flask框架提供接口服务，MongoDB用于保存多平台舆情数据及多模态分析结果。根据系统功能和业务流程，整体架构可划分为前端展示层、后端服务层、数据采集层、多模态分析层和数据存储层五个部分。
数据采集层由微博、抖音、百度贴吧和小红书四个平台爬虫组成。不同平台的采集入口和采集策略各异，但均通过统一运行器进行调度，并将采集结果整理为统一字段结构后写入MongoDB对应集合。
数据存储层以MongoDB为核心，按平台集合保存舆情数据和分析结果，同时使用JSON文件保存运行配置、平台配置、账号信息、采集源申请和个人追踪记录。该架构既符合当前项目实际功能，也为后续平台扩展和分析引擎切换提供了基础。
多模态分析层主要负责对舆情正文、图片和OCR文字进行分析，提取文本金融属性、风险要素、图像特征和融合评分结果，最终形成综合风险分、风险等级、风险原因和处置建议。
后端服务层由Flask后端程序承担，主要负责登录认证、角色权限控制、接口转发、数据查询、采集调度、分析调度和系统配置维护等任务。前端页面发出的请求通过后端接口进行处理，后端再根据业务需要访问MongoDB、读取配置文件、调用爬虫运行器或启动多模态分析模块，从而实现前后端之间的数据交互。
前端展示层是系统与用户直接交互的部分，主要包括监测首页、舆情监测、风险分析、详情报告、个人追踪以及后台管理相关页面。普通用户通过前台业务端查看舆情数据、筛选风险内容、查看分析结果和详情报告，并可对重点舆情进行个人追踪；管理员则通过后台管理端完成平台管理、采集任务管理、模型管理、运行日志查看、账号管理、用户管理和系统设置等操作。前台业务端与后台管理端的区分，使系统能够避免普通用户接触复杂配置和运行维护功能，不同角色的操作边界更加清晰。

3.2.2 系统模块设计
根据系统总体架构和详细设计内容，本系统可划分为多元舆情采集模块、MongoDB数据库模块、文本风险识别模块、图像内容检测模块、多模态融合风险评估模块、后端接口模块和前端页面模块。各模块围绕"数据采集—数据存储—风险分析—结果展示"的业务流程相互配合，共同完成网络金融舆情风险监测任务。
多元舆情采集模块是系统的数据入口，面向微博、抖音、百度贴吧和小红书四个平台。该模块根据不同平台特点采用差异化采集策略，完成关键词检索、帖子解析、图片下载、关键词过滤和去重判断，并将有效舆情数据写入对应数据库集合，为后续风险分析提供原始数据。
MongoDB数据库模块负责保存系统采集到的舆情数据及分析结果。由于不同平台的原始字段存在差异，系统在入库前将数据整理为统一字段结构——平台来源、作者、正文、发布时间、图片路径、关键词、分析状态和分析结果等。数据库按平台分集合存储，既便于数据隔离和维护，也便于后端统一读取和前端集中展示。
文本风险识别模块主要对舆情正文和OCR识别出的图片文字进行分析，识别金融属性、风险要素、情感倾向和煽动性特征。
图像内容检测模块负责对帖子配图进行二维码检测、模糊度检测、色彩丰富度分析、设计感评估和OCR文字提取，用于发现隐藏在图片中的风险线索。
多模态融合风险评估模块在文本分析和图像检测结果的基础上，综合正文、图片和OCR文本三类特征，计算综合风险分、划分风险等级，并生成风险原因、分析结论和处置建议。该模块的输出结果写回MongoDB，并作为前端风险展示和详情报告的数据来源。
后端接口模块基于Flask实现，负责用户登录权限、数据查询、采集任务调度、分析任务调度、平台配置、账号管理和系统设置等功能。
前端页面模块负责系统交互与结果展示，普通用户主要查看监测首页、舆情监测、风险分析和详情报告，管理员则通过后台页面维护平台配置、采集任务、运行日志、账号和系统参数。通过上述模块划分，系统结构更加清晰，也便于后续实现、测试和维护。

3.2.3 系统部署结构设计
本系统采用本地化部署方式，整体由前端静态页面、Flask后端服务、MongoDB数据库、多平台爬虫模块和多模态分析模块组成。前端页面文件由Flask后端统一提供访问，用户通过浏览器进入系统并完成登录、舆情查看、风险分析和后台管理等操作。后端服务主要负责接口响应、权限校验、采集任务调度、分析任务调度、数据查询和系统配置维护。MongoDB部署在本地环境中，用于保存微博、抖音、百度贴吧、小红书四个平台的舆情数据及其风险分析结果。
爬虫模块位于项目后端相关目录下，由管理员在后台采集任务管理中统一调用。各平台爬虫采集到的图片保存在对应平台的本地图片目录中，MongoDB中主要保存图片路径，便于后续前端预览和多模态分析调用。多模态分析模块与后端部署在同一项目中，可根据系统设置选择API版、本地版或端到端版分析文件。系统运行配置、平台配置、账号信息、采集申请和个人追踪记录等通过JSON文件保存。该部署结构适合毕业设计实验环境，层次清晰，便于调试和维护，但未设计分布式部署或生产级集群。
3.3 详细设计
3.3.1 多元舆情采集模块设计
多元舆情采集模块是本系统获取网络金融舆情数据的基础模块，同时也是后续风险分析、前端展示和详情报告生成的数据来源。网络金融舆情分布在不同互联网平台中，各平台的数据入口、页面结构、内容形式和采集规则存在差异，系统在设计时采用了"差异化采集、统一化调度、标准化入库"的思路。采集模块面向微博、抖音、百度贴吧和小红书四个平台，各平台爬虫根据自身特点独立实现，但统一由爬虫运行器进行调度，并最终转换为相对一致的数据结构写入MongoDB。系统通过crawlers/runner.py维护平台ID、中文名称、爬虫目录、入口文件和MongoDB集合映射，从而实现多平台采集任务的统一加载与运行。
在系统操作流程上，采集模块主要由管理员在后台使用。管理员可在平台管理中维护不同平台的采集规则——关键词、Cookie、采集页数、采集天数、是否要求图片等参数均在可配置范围内；在采集任务管理中，管理员可选择需要采集的平台并启动、暂停或继续采集任务。普通用户不直接控制采集过程，只能在前台查看采集后的舆情结果或提交新增采集源申请。采集任务运行时，系统会记录各平台的运行状态和日志信息，并统计写入、跳过、未命中、丢弃等结果，便于管理员判断采集任务执行情况。
在平台采集策略方面，微博爬虫主要依据关键词和时间范围进行搜索——系统根据管理员配置的金融风险相关关键词对微博搜索结果进行抓取，解析微博卡片中的作者、正文、发布时间、话题、图片链接和命中关键词等信息，并根据图片规则下载相关图片。
抖音采集模块围绕关键词采集图文作品，重点获取作品文本、作者信息、发布时间、图片内容及相关标识。抖音内容形式较为多样，系统在采集时需根据图文内容要求进行筛选，保证进入后续分析流程的数据具有文本和图片基础。
百度贴吧采集模块以吧名作为发现入口——股票吧、基金吧、金融吧、证券吧、期货吧、外汇吧、银行吧、理财吧等相关贴吧均在采集范围内。系统先抓取主题帖列表，再进入具体帖子页面解析正文、作者、发布时间和图片等内容。为避免单一贴吧数据过于集中，系统支持对多个贴吧进行随机轮换采集，并在遇到不可访问、内容异常或平台限制等情况时跳过异常对象，不中断整体采集任务。
小红书采集模块主要按关键词搜索图文笔记，解析笔记标题、正文、作者、标签、图片等信息，并将其转换为系统统一的舆情字段结构，便于与其他平台数据共同展示和分析。
从具体采集过程来看，各平台虽然入口不同，但业务流程基本一致：系统先根据平台配置构造采集任务并请求目标平台数据；随后解析帖子或笔记内容，提取平台来源、作者昵称、正文、发布时间、话题标签、图片链接、关键词和匹配关键词等信息；对于满足条件的图片，系统下载到本地目录并在数据库中保存图片路径；之后进行关键词过滤和重复数据判断，剔除无关内容和已采集内容；最终将有效数据写入MongoDB中对应的平台集合。微博、抖音、百度贴吧和小红书分别对应不同集合，但最终字段结构尽量统一，以便前端和分析模块复用。
采集模块与多模态分析模块之间采用"先入库、后分析"的衔接方式。采集任务完成后，原始舆情数据先写入MongoDB，随后由系统自动分析流程或管理员手动补分析流程对未分析且包含文本和图片的数据进行多模态风险识别。分析完成后，风险结果再写回MongoDB，并供前端监测首页、舆情监测页面、风险分析页面和详情报告页面调用。通过该设计，采集模块不仅承担数据获取任务，也为系统后续的文本识别、图片检测、OCR分析和融合评分提供稳定的数据基础。


3.3.2 MongoDB数据库设计
本系统选择MongoDB作为主要舆情数据存储数据库，核心原因在于系统采集的数据来源复杂、字段结构具有一定差异性。微博、抖音、百度贴吧和小红书等平台在帖子标识、作者信息、正文内容、图片组织方式和发布时间等方面并不完全一致，系统还需保存图片路径、关键词、OCR结果、图像检测结果和多模态分析结果等半结构化数据。相比固定表结构的关系型数据库，MongoDB的文档型存储方式更适合保存字段灵活、结构可扩展的多平台舆情数据。
在数据库结构上，本系统默认使用test数据库，并按平台划分为四个主要集合：微博数据存入test_weibo，抖音数据存入test_douyin，百度贴吧数据存入test_baidu，小红书数据存入test_xhs。按平台分集合存储，一方面便于平台数据隔离和单独维护，另一方面也便于后端在前端查询时进行统一聚合。系统中的统一爬虫运行器也维护了平台ID、入口文件和MongoDB集合之间的映射关系，使采集结果能够准确写入对应集合。
在文档字段设计方面，系统尽量将不同平台采集到的数据转换为统一结构。单条舆情文档主要包括id、_key、note_id、platform、platform_name、screen_name、text、created_at、source、keyword、matched_keywords、topics、pics、crawl_time、inserted_at、analysis、analysis_status、analysis_error、analysis_image、analysis_at等字段。其中id用于表示平台内的唯一帖子标识，_key可作为去重辅助字段，note_id用于兼容小红书相关历史字段；platform和platform_name用于区分数据来源；screen_name、text、created_at分别保存作者昵称、正文内容和发布时间；pics保存本地图片路径列表；analysis用于保存多模态分析结果，analysis_status用于记录分析状态——已完成、分析中、错误或无图片等。
在图片与前端访问方面，数据库中不直接保存图片二进制内容，而是保存图片本地路径。后端在序列化MongoDB文档时，会将_id转换为字符串，补齐id、platform_name、screen_name等字段，并将pics转换为包含路径和访问地址的结构，使前端能够通过后端媒体接口预览图片。这样既减少了数据库存储压力，也便于多模态分析模块根据本地路径读取图片。
在数据去重设计方面，系统需避免相同舆情被重复采集和重复展示。采集模块通过各平台的seen_ids文件记录已采集内容，同时在MongoDB中结合id、_key、note_id等字段进行重复判断。对于历史数据兼容问题，后端还提供索引清理、缺失字段补齐和错误集合迁移等逻辑——清理历史不兼容索引、补齐_key和note_id，以及根据platform字段将误入集合的数据迁移到正确集合中。该设计有助于提高数据库数据的一致性和可维护性。
3.3.3 文本风险识别模块设计
文本风险识别模块是多模态风险分析中的基础模块，主要用于对舆情正文以及图片OCR识别出的文字进行金融风险特征提取。网络金融舆情中的风险信息往往首先通过文字表达——保本承诺、高收益诱导、拉人头返利、内幕荐股、私域引流、数字货币骗局、非法外盘交易、贷款征信诈骗、养老金融骗局等内容均属此列。模块不仅要处理采集到的正文文本，还需对OCR提取出的图片文字再次进行分析，以提高系统对图文混合风险内容的识别能力。
从模块输入来看，文本风险识别模块的输入主要包括两类：一类是微博、抖音、百度贴吧、小红书等平台采集到的舆情正文；另一类是图像内容检测模块通过OCR从图片中提取出的文字。
从模块输出来看，文本风险识别结果主要包括金融属性分类、风险要素识别、情感倾向分析和煽动性评估四个方面。金融属性分类用于判断文本属于哪类金融舆情——理财、投资、贷款、数字货币等——从而确认其是否具备进一步风险分析价值。风险要素识别用于提取文本中的具体风险类别、命中词汇和出现数量，如"保本""稳赚不赔""返利""内幕消息""限时名额"等词汇或表达。
情感倾向分析主要判断文本表达的情绪特征，辅助识别是否存在过度夸张、强烈诱导或情绪化宣传等情况。煽动性评估则进一步分析文本中的紧迫感、收益诱导、稀缺性暗示和行动号召等表达——"马上加入""最后机会""已有多人参与"等内容均可作为判断诱导程度的依据。
在系统流程中，文本风险识别模块的分析结果会作为多模态融合评分的重要输入。系统后续会结合正文文本风险、图片视觉风险和OCR文本风险，计算综合风险分和风险等级，并生成风险原因、结论和处置建议。文本分析结果还会写入MongoDB中的分析字段，并在前端详情报告中展示，使用户能够查看金融属性、风险类别、风险词汇、情感结果和煽动性结果等内容。

3.3.4 图像内容检测模块设计
图像内容检测模块是系统多模态风险分析的重要组成部分，主要用于识别舆情配图中可能存在的金融风险信息。网络金融舆情并不总是以纯文本形式传播——部分风险内容会以收益海报、产品宣传图、聊天截图、二维码、联系方式等形式出现在图片中。比如，一些帖子正文可能只包含简短描述，而图片中却包含"保本""稳赚""高收益""扫码咨询"等营销话术或引流入口。如果系统只分析正文文本，容易遗漏这类隐藏在图片中的风险线索。系统需要在文本风险识别之外单独设计图像内容检测模块，对帖子配图进行补充分析。
从模块输入来看，图像内容检测模块主要接收舆情文档中保存的本地图片路径。采集模块在下载图片后会将图片路径写入MongoDB文档，后续分析流程即可根据该路径读取图片进行检测。
从模块输出来看，图像内容检测结果主要包括二维码检测结果、图片模糊度分数、色彩丰富度分数、设计感评分、OCR文字及其位置信息等。二维码检测用于判断图片中是否存在私域引流入口——在网络金融风险场景中，二维码常被用于将用户引导至外部社群、个人账号或交易渠道，其存在情况可作为风险评估的重要参考。图片模糊度检测用于评估图片清晰程度，保证后续OCR识别和人工查看具有一定可读性。色彩丰富度和设计感评估则用于辅助判断图片是否具有明显营销海报特征——高饱和配色、突出收益数字、强引导性排版等视觉宣传特点。
OCR文字识别是图像内容检测模块中与文本分析衔接最紧密的部分。系统通过OCR提取图片中的文字及其位置信息，将图片中的宣传语、联系方式、金额数字、收益承诺和扫码提示等内容转化为可分析文本，并再次进入文本风险识别模块进行二次分析。系统会对OCR文本执行金融属性判断、风险要素识别、情感倾向分析和煽动性评估，从而形成OCR文本风险结果。该结果与正文文本风险、图片视觉特征共同参与多模态融合评分，用于计算综合风险分和风险等级。
图像内容检测结果最终写入MongoDB文档中的分析结果字段，并在前端舆情详情报告中展示。用户在查看单条舆情报告时，可以同时看到图片检测、OCR文字、风险原因和多模态融合结果，从而更直观地理解系统判断该舆情存在风险的依据。通过该模块，系统能够弥补单纯文本分析的不足，提高对图文混合金融风险舆情的识别能力，并为后续综合风险评分提供更加完整的特征来源。

3.3.6 后端接口设计
后端接口是连接前端页面、数据库、采集模块和多模态分析模块的核心纽带。本系统后端基于Flask实现，既负责提供前端页面访问，也负责处理前端发起的业务请求。根据系统功能划分，后端接口主要承担用户登录权限控制、舆情数据查询、采集任务调度、多模态分析调度、平台配置管理、个人追踪、账号管理和系统设置等任务。系统采用前台业务接口与后台管理接口相区分的设计思路——普通用户主要调用舆情查看、风险分析、详情报告和个人追踪相关接口，管理员则调用采集任务、平台配置、运行日志、账号管理和系统设置等后台接口。
在基础页面与系统状态接口方面，后端需提供主页面、详情报告页面、采集申请处理页面和新增账号页面等访问入口，同时提供健康检查接口用于返回服务运行状态、数据库连接状态等信息。该类接口主要保证用户能够正常进入系统页面，也便于管理员判断后端服务是否处于可用状态。
在用户登录与账号相关接口方面，系统需提供登录、注册、会话获取、个人资料修改和密码修改等接口。普通用户登录后进入前台业务端，管理员登录后进入后台管理端。后端在处理请求时需识别用户角色，并根据角色控制可访问的接口范围。普通用户的主要操作包括查看舆情、提交采集源申请、加入个人追踪和维护个人信息；管理员则需具备后台维护权限。普通用户不能启动或暂停爬虫、修改采集源、查看后台运行日志或管理账号。
在舆情数据接口方面，后端需提供舆情列表查询、单条舆情详情查询和图片媒体访问接口。前端舆情监测页面通过列表接口获取多平台舆情数据，并根据平台、关键词、风险等级、分析状态和图片条件进行筛选；详情报告页面则通过单条数据接口读取指定舆情的完整信息。由于MongoDB文档中包含ObjectId、本地图片路径和平台字段等后端数据格式，接口返回前需进行序列化处理——将ObjectId转换为字符串，补齐平台名称、作者昵称等信息，并将图片路径转换为前端可访问的媒体地址，保证前端通过统一格式展示不同平台数据。
在采集任务接口方面，后端需提供采集任务启动、暂停和状态查询接口。管理员在后台采集任务管理页面选择平台后，后端根据所选平台创建采集任务，并通过统一爬虫运行器调度微博、抖音、百度贴吧和小红书爬虫。采集状态接口用于返回任务运行状态、日志和统计信息——写入数量、跳过数量、未命中数量和丢弃数量等。此类接口属于后台管理接口，应受到管理员权限校验保护，普通用户不能直接调用。
在多模态分析接口方面，后端需支持未分析数据补分析功能。管理员可在后台触发补分析操作，系统从MongoDB中筛选未分析且符合条件的数据，调用当前配置的多模态分析引擎进行处理，并将分析结果写回文档。系统还提供手动上传图文分析入口，用于对单条上传内容进行分析。分析完成后，前端即可根据分析状态、综合风险分和风险等级进行展示。
在平台配置与采集源申请接口方面，后端需支持平台配置读取与保存、采集源列表获取、采集源提交、采集申请详情查看和管理员审核等功能。普通用户只能提交采集源申请，不能直接修改系统采集平台；管理员可在后台查看申请详情，补充采集规则后决定通过或退回。该接口设计体现了前台业务操作和后台管理操作的边界。
在个人追踪接口方面，系统需提供追踪列表获取、加入追踪和取消追踪功能。普通用户可将重点舆情加入个人追踪列表，后端按用户维度保存追踪记录，避免不同用户之间的数据混淆。该模块与舆情详情和舆情列表接口相配合，使用户能够持续关注重点风险内容。
在后台管理接口方面，系统需提供账号管理、用户管理、采集规则查看和系统设置等接口。管理员可通过账号管理接口新增账号、修改账号信息、启用或禁用账号、重置密码；通过用户管理接口查看角色权限和用户操作记录；通过系统设置接口维护分析引擎文件、分析并发数、采集目标量、自动分析开关等运行参数，并在需要时重置分析引擎。
总体来看，后端接口设计以统一JSON数据交互为基础，前端通过fetch调用接口获取或提交数据，后端根据用户角色、业务类型和数据状态完成相应处理。前台业务接口与后台管理接口的分离，使系统既能满足普通用户查看舆情结果、风险分析和详情报告的需求，也能满足管理员维护采集任务、平台配置、日志、账号和系统参数的需求。该设计与系统前后台分权的总体结构一致，有利于提高系统的可维护性和使用安全性。

3.3.7 前端页面设计
前端页面是系统面向用户展示舆情数据、风险分析结果和后台管理功能的重要部分。本系统前端采用HTML、CSS、JavaScript实现，整体呈现为单页面式交互结构，通过左侧导航栏组织不同功能页面，并根据用户角色区分前台业务端和后台管理端。普通用户登录后主要进入前台页面，管理员登录后可看到更多后台管理入口。该设计遵循"普通用户看结果，管理员管配置"的原则，使普通用户能够专注于舆情查看和风险研判，管理员则负责采集配置、运行维护和系统管理。
前台业务端主要包括监测首页、数据采集申请、舆情监测、个人追踪、风险分析和个人设置等页面。监测首页用于展示系统总体运行和舆情风险概况——舆情总量、风险等级分布、平台来源分布以及最新中高风险或高风险舆情——使用户进入系统后能够快速了解当前监测情况。数据采集申请页面面向普通用户，用户可提交新的采集源申请并填写平台名称、目标网址、平台类型和关键词等基本信息，但不能直接修改系统采集配置。
舆情监测页面是普通用户使用频率较高的页面，主要以列表形式展示采集和分析后的舆情数据。列表中突出平台来源、作者昵称、正文摘要、图片、发布时间、风险分数、风险等级和分析状态等信息。为便于用户从大量舆情中快速定位目标内容，页面支持平台、关键词、风险等级、情感倾向、分析状态、是否含图等筛选条件。对于较长的帖子正文，页面采用默认显示摘要、点击后展开或收起的方式，既保证列表整洁又方便用户查看完整内容。用户点击某条舆情后可进入详情报告页面，查看原文、图片、风险原因、文本分析、图像检测、OCR分析、多模态融合结果和处置建议。
个人追踪页面用于展示用户已关注的重点舆情——普通用户在舆情监测或详情查看过程中可将需要持续关注的帖子加入个人追踪列表，后续在该页面集中查看。风险分析页面主要展示多模态风险分析结果，包括风险分类分布、情感分布以及中风险、中高风险、高风险等不同等级内容，帮助用户从风险角度对舆情数据进行集中研判。个人设置页面则用于普通用户维护个人资料和修改密码。
后台管理端面向管理员，页面包括平台管理、采集任务管理、模型管理、运行日志、账号管理、用户管理和系统设置等。平台管理页面用于维护内置平台和扩展平台，并处理普通用户提交的采集源申请——管理员可查看申请详情，补充采集规则后决定通过或退回。采集任务管理页面用于选择平台、启动或暂停采集任务、查看采集规则和任务统计，并提供历史数据补分析入口。模型管理页面用于展示分析引擎及相关组件状态，运行日志页面用于集中查看采集和分析过程中的日志信息。账号管理和用户管理页面用于维护账号信息、角色权限和用户操作记录。系统设置页面用于维护分析引擎文件、分析并发数、自动分析开关、单轮采集目标量等运行参数。
在前后端交互方面，前端主要通过JavaScript的fetch方法调用Flask后端提供的JSON接口，获取舆情列表、详情数据、风险分析结果、追踪记录和后台配置等信息。后端返回数据前会将MongoDB文档进行序列化处理，使平台名称、图片路径、分析状态等字段能够被前端直接展示。图片文件通过后端/media接口访问，前端根据返回的图片地址在列表和详情报告中显示配图。通过上述页面设计，系统能够在浏览器端完成舆情查看、风险筛选、详情报告和后台管理等主要操作，并保持前台业务和后台维护之间的清晰边界。
3.4 本章小结
本章在第二章系统分析的基础上，对网络金融舆情风险监测系统的整体设计方案进行了阐述。明确了系统的设计目标——建设一个面向金融监管人员、平台管理人员和舆情监测人员的多平台舆情风险监测系统，实现舆情采集、数据存储、风险分析、结果展示、详情报告、个人追踪和后台管理等功能。系统设计遵循实用性、模块化、可扩展性、可维护性、安全性和一致性等原则，以保证系统结构清晰、功能边界明确，并便于后续维护和扩展。
概要设计部分对系统总体架构、功能模块、核心业务流程和部署结构进行了设计。系统采用B/S架构，整体划分为前端展示层、后端服务层、数据采集层、多模态分析层和数据存储层。前端展示层负责舆情结果和风险分析结果的可视化呈现，后端服务层负责接口调用、权限校验、采集调度和分析调度，数据采集层负责微博、抖音、百度贴吧、小红书等平台的数据获取，多模态分析层负责文本、图片和OCR内容的综合分析，数据存储层则通过MongoDB和配置文件保存舆情数据、分析结果及系统运行信息。
详细设计部分围绕系统核心模块展开——多元舆情采集模块、MongoDB数据库设计、文本风险识别模块、图像内容检测模块、多模态融合风险评估模块、后端接口设计和前端页面设计。系统通过统一爬虫运行器调度不同平台采集任务，将多平台数据转换为统一字段结构后写入MongoDB；通过文本分析、图片检测、OCR识别和多模态融合评分生成综合风险分、风险等级、风险原因和处置建议；再通过前端页面完成舆情监测、风险分析和详情报告展示。
本章还强调了前台业务端与后台管理端分离的设计思路。普通用户主要查看舆情结果、进行筛选检索、查看风险报告、提交采集源申请和维护个人追踪；管理员则负责平台配置、采集任务控制、运行日志查看、模型状态管理、账号管理和系统参数维护。

第4章  系统实现
4.1  开发环境与技术栈
本系统基于Windows开发环境实现，后端采用Python与Flask框架构建接口服务，使用MongoDB存储多平台舆情数据及分析结果；前端采用HTML、CSS、JavaScript实现交互界面。系统集成微博、抖音、百度贴吧、小红书等爬虫模块，并结合PyTorch、TensorFlow、OpenCV、Transformers等技术完成文本识别、图像检测和多模态风险分析。
4.2  多平台舆情数据采集与存储实现
本系统的多平台舆情采集与存储功能按照"平台独立实现、后端统一调度、数据分集合存储、前端可视化维护"的思路进行开发。代码层面，系统在整合/crawlers目录下分别为微博、抖音、百度贴吧和小红书建立独立爬虫模块，每个平台保留自己的配置文件、入口文件、图片保存目录和seen_ids.txt记录文件。这样做的好处是将不同平台的登录Cookie、搜索入口、页面结构、解析逻辑和异常处理分开维护，避免某个平台规则变化波及其他平台。
为统一调用各平台爬虫，系统在crawlers/runner.py中维护平台映射关系，将平台标识、中文名称、爬虫目录、入口文件和MongoDB集合名称对应起来。管理员在后台"采集任务管理"中选择平台并启动采集后，backend/app.py中的CrawlTask会接收平台列表和采集参数，再调用runner.py动态加载对应爬虫入口。运行器加载爬虫时会临时切换导入上下文，避免不同平台目录下同名setting.py相互污染，同时将运行时Cookie、关键词、页数、是否要求图片等配置传递给对应爬虫。
在采集策略上，系统支持多平台并行运行，并通过CRAWL_TOTAL_TARGET将总采集目标量近似分配给各个平台。微博和小红书主要按关键词搜索图文内容，百度贴吧以吧名和页面为入口获取主题帖，抖音按关键词采集图文作品。采集过程中，爬虫会提取平台来源、帖子ID、作者昵称、正文内容、发布时间、匹配关键词、话题标签和图片地址等字段，并将图片下载到本地目录。MongoDB中主要保存图片路径和相对访问地址，而非直接保存图片二进制内容。
数据存储采用MongoDB，按平台分集合保存——微博写入test_weibo，抖音写入test_douyin，百度贴吧写入test_baidu，小红书写入test_xhs。后端通过platform_collection、iter_platform_collections、normalize_documents_collections等函数维护集合映射和数据归属，避免不同平台数据混入同一集合。为适配前端统一展示，各平台数据在入库前会尽量整理为一致字段——id、platform、platform_name、screen_name、text、created_at、pics、matched_keywords、analysis_status等。
去重方面，系统同时使用本地seen_ids.txt和MongoDB字段进行控制。爬虫会先根据平台帖子ID判断是否已采集，入库时再结合id、_key、note_id等字段避免重复写入。后端还实现了历史索引清理和缺失字段补齐逻辑，用于解决项目整合后可能出现的唯一索引冲突、空字段重复和集合归属错误等问题。

为方便管理员检查采集结果，系统还在后台加入了数据库管理功能。该功能参考MongoDB Compass的交互风格，在前端以资源管理器形式展示localhost:27017、数据库和集合层级，并在集合右侧显示当前数据数量。管理员选择集合后，可通过Filter输入JSON查询条件，使用Reset、Find、分页、刷新、上一页和下一页等控件浏览文档。每条数据以字段树形式展示，嵌套对象和数组可展开或隐藏。系统还支持Add Data、新增文档、编辑文档、删除文档、清空集合等维护操作。
数据库管理接口由/api/admin/database/*提供，所有写操作均需管理员权限。由于MongoDB文档中可能包含ObjectId、datetime、Binary等不能直接序列化为JSON的BSON类型，后端在app.py中实现了bson_to_jsonable方法将其转换为前端可展示结构。系统对admin、config、local等系统库进行保护，删除集合或清空数据前需二次确认。整体来看，该模块既实现了多平台数据采集和统一存储，也提供了面向管理员的数据核查与维护能力。

4.4  风险预警展示平台实现
本风险预警展示平台主要由前端页面、后端舆情接口和MongoDB分析结果共同支撑。系统在完成舆情采集和多模态分析后，会将综合风险分、风险等级、文本特征、图像特征、OCR信息和融合分析结果写回MongoDB。前端通过调用/api/items接口获取多平台舆情数据，并在业务端页面中进行统一展示。后端在返回数据前会通过serialize_doc对文档结构进行整理，补齐平台名称、帖子ID、用户名、图片访问地址等字段，使微博、抖音、百度贴吧和小红书的数据能够按照相同结构渲染。
在前端实现上，系统采用HTML、CSS和JavaScript构建单页面应用，通过左侧导航在首页、舆情监测、个人追踪、风险分析和详情报告等页面之间切换。舆情监测页面是风险预警展示的主要入口，页面以帖子卡片形式展示舆情内容——每条卡片包含平台来源、作者昵称、发布时间、正文内容、图片、风险等级和综合风险分。为避免长文本影响阅读，正文默认只显示前几行，用户可点击展开按钮查看全文。系统还根据用户名首字生成头像，使列表展示更加直观。
风险筛选功能主要在前端app.js中实现。用户可通过平台按钮快速筛选微博、抖音、贴吧和小红书数据，也可通过中风险、中高风险、高风险和全部等条件查看不同风险等级的内容。系统根据分析结果中的综合风险分计算风险等级，并使用不同颜色和标签进行提示。对于已完成分析的数据，前端展示风险分和分析状态；对于未分析或缺少图片的数据，则显示相应状态，避免用户误以为系统异常。
风险分析页面进一步对预警结果进行汇总展示——平台来源分布、风险等级分布、情感倾向统计和中高风险内容列表等。系统将预警展示和风险分析做了区分：舆情监测页面强调实时浏览和筛选，风险分析页面强调统计概览和重点风险内容查看。用户点击"查看详情"后可进入独立报告页面，查看原文、图片、文本风险识别结果、图像检测结果、OCR识别结果、融合评分、风险原因和处置建议。
系统还提供个人追踪功能。用户在舆情监测页面点击"加入追踪"后，该帖子会写入watchlist.json并在"个人追踪"页面中单独展示。这样用户可对重点舆情进行持续关注，而不需要每次重新筛选。整体来看，风险预警展示平台实现了从风险数据读取、列表展示、条件筛选、统计分析、详情报告到个人追踪的完整展示流程，为网络金融舆情风险监测提供了直观的业务操作界面。


4.5  后端功能实现
本系统后端采用Flask框架实现，核心代码集中在整合/backend/app.py中。后端主要承担前端页面服务、REST接口提供、MongoDB数据访问、多平台爬虫调度、多模态分析调用、账号权限控制和后台管理等功能。程序启动时会读取runtime_config.json、platform_settings.json、collection_sources.json、users.json等配置文件，将分析引擎、并发数量、自动分析开关、平台Cookie、关键词、账号和采集源申请等信息加载到系统中，并在配置变更后写回本地文件，实现轻量化持久化管理。
在数据访问方面，后端通过PyMongo连接MongoDB，并将微博、抖音、百度贴吧和小红书分别映射到test_weibo、test_douyin、test_baidu、test_xhs四个集合。代码中封装了platform_collection、iter_platform_collections、serialize_doc等函数，用于统一定位平台集合、遍历多平台数据和整理返回给前端的文档结构。由于MongoDB中存在ObjectId、datetime、Binary等不能直接JSON序列化的数据类型，后端实现了bson_to_jsonable方法将特殊BSON类型转换为前端可识别的字符串或对象结构，从而保证普通舆情页面和数据库管理页面都能稳定显示。
采集调度功能由CrawlTask和WebCrawlController负责。管理员在后台选择采集平台并启动任务后，后端根据平台列表调用crawlers/runner.py，由统一运行器动态加载各平台爬虫入口并传入Cookie、关键词、页数、图片规则等配置。采集任务运行过程中支持暂停、继续、状态查询和日志返回，前端通过/api/crawl/status轮询展示写入数量、重复数量、未命中数量和丢弃数量等统计信息。
多模态分析功能采用可插拔设计。后端根据runtime_config.json中的analysis_engine_file动态加载MultimodalRiskFusion分析类，可在API版、本地版和端到端深度网络版之间切换。采集数据会先写入MongoDB，再根据auto_analyze_crawled_items配置决定是否自动分析，也可由管理员调用/api/analyze/unprocessed对历史未分析数据进行补分析。分析时系统使用ThreadPoolExecutor按analysis_max_workers并行处理多条帖子，分析完成后将analysis、analysis_status、analysis_at等字段更新回数据库。
权限控制方面，系统通过登录接口区分普通用户和管理员。普通用户主要访问舆情监测、风险分析、个人追踪和采集申请功能；管理员则可进入后台管理端。后端通过require_admin对采集控制、平台管理、账号管理、系统设置和数据库管理等接口进行校验，避免普通用户误操作核心配置。整体来看，后端不仅负责接口转发，还承担了数据统一、任务调度、并发分析、权限隔离和异常兼容等关键职责。


4.6  前端功能实现
本系统前端采用原生HTML、CSS和JavaScript实现，主要文件包括整合/frontend/index.html、app.js和styles.css。前端整体采用单页面应用的组织方式，通过左侧导航栏和data-view属性切换不同功能页面，无需频繁跳转完整页面。index.html负责定义页面结构，styles.css负责界面样式和响应式布局，app.js负责数据请求、状态管理、页面渲染和用户交互逻辑。
在页面结构上，前端分为前台业务端和后台管理端。前台业务端主要包括首页、数据采集、舆情监测、个人追踪、风险分析和个人设置等页面；后台管理端主要包括平台管理、采集任务管理、模型管理、运行日志、数据库管理、账号管理、用户管理和系统设置等页面。系统根据当前登录用户的角色控制导航栏显示——普通用户只能访问前台业务功能，管理员才能进入后台管理端。
舆情监测页面是前端展示的核心部分。页面通过/api/items接口获取后端返回的多平台舆情数据，并以卡片形式展示平台来源、用户昵称、发布时间、正文内容、图片、风险等级和综合风险分。为提高阅读体验，长文本默认只展示前几行，点击按钮后可展开全文。用户头像由用户名首字生成，既避免依赖外部头像资源，也能提升列表辨识度。页面还支持按平台、风险等级、分析状态和时间范围进行筛选，使用户能够快速定位重点舆情。
风险分析和详情报告用于展示更细粒度的分析结果。风险分析页面对平台分布、风险等级、情感倾向和中高风险内容进行汇总，帮助用户从整体上了解当前舆情风险情况。用户点击"查看详情"后会进入report.html页面，前端通过单条数据接口读取帖子详情，并展示原文、图片、文本风险识别、图像检测、OCR识别、多模态融合评分和处置建议等内容。个人追踪功能则允许用户将重点帖子加入追踪列表，并在个人追踪页面集中查看。
后台管理端面向管理员，主要用于系统维护。平台管理页面支持查看采集申请和配置平台参数；采集任务管理页面支持选择平台、启动采集、暂停采集、查看任务统计和运行状态；系统设置页面可配置分析引擎、并发数、自动分析开关和平台Cookie。新增的数据库管理页面参考MongoDB Compass风格，左侧以资源管理器展示数据库和集合层级，右侧支持JSON条件查询、分页浏览、字段树展开、新增文档、编辑文档和删除文档等操作。整体来看，前端不仅完成了舆情风险结果展示，也承担了采集配置、系统管理和数据库维护等交互功能。


4.7  本章小结
本章在前文系统分析和系统设计的基础上，对网络金融舆情风险监测系统的具体实现过程进行了说明。系统在Windows本地环境下开发与运行，项目主体位于"整合"目录，整体采用B/S架构。后端使用Python和Flask框架实现接口服务、权限控制、数据查询、采集任务调度和多模态分析调度；数据库使用MongoDB存储微博、抖音、百度贴吧、小红书等平台的舆情数据和分析结果；前端采用HTML、CSS和JavaScript实现页面展示与交互；同时结合多平台爬虫模块和多模态分析模块支撑系统核心业务流程的运行。
后端以backend/app.py为核心，负责提供页面访问、REST接口、用户登录权限、MongoDB数据读写、爬虫任务控制、分析引擎调度以及后台管理等功能。前端则以frontend/index.html、app.js和styles.css为主要文件，通过单页面式交互和左侧导航栏组织系统功能，并根据普通用户和管理员角色展示不同页面。
系统通过统一爬虫运行器调度微博、抖音、百度贴吧和小红书四个平台的采集程序，完成关键词检索、帖子解析、图片下载、关键词过滤、重复数据判断和MongoDB入库等操作。不同平台的数据最终被整理为统一字段结构并分别写入对应集合，为后续分析和展示提供数据基础。系统通过统一的MultimodalRiskFusion接口对舆情正文、本地图片和OCR文字进行综合分析，生成综合风险分、风险等级、风险原因、分析结论和处置建议，并将结果写回MongoDB。前端在监测首页、舆情监测页面、风险分析页面和详情报告页面中展示分析结果，实现了从舆情采集、数据存储、风险分析到前端展示和后台管理的完整流程。

在风险分析方面，系统结合正文文本、图片内容和OCR文字结果，完成金融属性判断、风险要素识别、情感倾向分析、煽动性评估、二维码检测和图像特征分析，并通过多模态融合方式生成综合风险分、风险等级、风险原因和处置建议。系统同时区分普通用户和管理员权限——普通用户主要查看舆情结果和详情报告，管理员负责平台配置、采集任务、日志、账号、系统设置和数据库管理等后台维护工作。
