4.2  多平台舆情数据采集与存储实现

本系统的多平台舆情采集与存储功能主要按照“平台独立实现、后端统一调度、数据分集合存储、前端可视化维护”的思路开发。代码层面，系统在整合/crawlers 目录下分别为微博、抖音、百度贴吧和小红书建立独立爬虫模块，每个平台保留自己的配置文件、入口文件、图片保存目录和 seen_ids.txt 记录文件。这样可以将不同平台的登录 Cookie、搜索入口、页面结构、解析逻辑和异常处理分开维护，避免某个平台规则变化影响其他平台。

为了统一调用各平台爬虫，系统在 crawlers/runner.py 中维护平台映射关系，将平台标识、中文名称、爬虫目录、入口文件和 MongoDB 集合名称对应起来。管理员在后台“采集任务管理”中选择平台并启动采集后，backend/app.py 中的 CrawlTask 会接收平台列表和采集参数，再调用 runner.py 动态加载对应爬虫入口。运行器加载爬虫时会临时切换导入上下文，避免不同平台目录下同名 setting.py 相互污染，同时将运行时 Cookie、关键词、页数、是否要求图片等配置传递给对应爬虫。

在采集策略上，系统支持多平台并行运行，并通过 CRAWL_TOTAL_TARGET 将总采集目标量近似分配给各个平台。微博和小红书主要按关键词搜索图文内容，百度贴吧以吧名和页面为入口获取主题帖，抖音按关键词采集图文作品。采集过程中，爬虫会提取平台来源、帖子 ID、作者昵称、正文内容、发布时间、匹配关键词、话题标签和图片地址等字段，并将图片下载到本地目录。MongoDB 中主要保存图片路径和相对访问地址，而不是直接保存图片二进制内容。

数据存储采用 MongoDB，并按照平台分集合保存。微博写入 test_weibo，抖音写入 test_douyin，百度贴吧写入 test_baidu，小红书写入 test_xhs。后端通过 platform_collection、iter_platform_collections、normalize_documents_collections 等函数维护集合映射和数据归属，避免不同平台数据混入同一集合。为了适配前端统一展示，各平台数据在入库前会尽量整理为一致字段，包括 id、platform、platform_name、screen_name、text、created_at、pics、matched_keywords、analysis_status 等。

去重方面，系统同时使用本地 seen_ids.txt 和 MongoDB 字段进行控制。爬虫会先根据平台帖子 ID 判断是否已采集，入库时再结合 id、_key、note_id 等字段避免重复写入。后端还实现历史索引清理和缺失字段补齐逻辑，用于解决项目整合后可能出现的唯一索引冲突、空字段重复和集合归属错误等问题。

为了方便管理员检查采集结果，系统还在后台加入数据库管理功能。该功能参考 MongoDB Compass，在前端以资源管理器形式展示 localhost:27017、数据库和集合层级，并在集合右侧显示当前 data 数量。管理员选择集合后，可以通过 Filter 输入 JSON 查询条件，使用 Reset、Find、分页、刷新、上一页和下一页等控件浏览文档。每条数据以字段树形式展示，嵌套对象和数组可以展开或隐藏。系统还支持 Add Data、新增文档、编辑文档、删除文档、清空集合等维护操作。

数据库管理接口由 /api/admin/database/* 提供，所有写操作均需要管理员权限。由于 MongoDB 文档中可能包含 ObjectId、datetime、Binary 等不能直接序列化为 JSON 的 BSON 类型，后端在 app.py 中实现 bson_to_jsonable 方法，将其转换为前端可展示结构。同时，系统对 admin、config、local 等系统库进行保护，删除集合或清空数据前需要二次确认。整体来看，该模块既实现了多平台数据采集和统一存储，也提供了面向管理员的数据核查与维护能力。
