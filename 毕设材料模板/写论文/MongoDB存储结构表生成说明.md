# 3.3.2 MongoDB 数据库存储结构表生成说明

## 使用位置

建议将该表放在论文第 3 章第 3.3.2 节“MongoDB数据库设计”中，用于说明系统舆情数据在 MongoDB 中的集合划分、字段结构和设计作用。

建议放置位置：在介绍“本系统采用 MongoDB 作为主要舆情数据存储数据库，四个平台按集合分开存储，但字段结构尽量统一”之后插入。

## 表格标题

建议表题为：

**表3-X  MongoDB 舆情数据存储结构设计**

其中 X 按论文第 3 章表格实际顺序填写。

## 格式要求

1. 表题放在表格上方，居中显示。
2. 表格按章节编号，例如“表3-1”“表3-2”。
3. 建议采用三线表形式，不使用竖线。
4. 表头居中，字段说明可左对齐。
5. 字体建议使用宋体五号；若表格较长，可使用小五号。
6. 表格内容应简明，不要把分析结果 JSON 全部展开。
7. 不展示真实 Cookie、账号、接口密钥、图片绝对路径等敏感信息。
8. 如果一张表过长，可以拆成两张表：一张展示集合设计，一张展示字段结构。

## 推荐表格方案

为了论文排版清晰，建议生成两张表：

1. **平台集合设计表**：说明不同平台数据分别存入哪个集合。
2. **统一字段结构表**：说明每条舆情文档的主要字段及用途。

如果论文篇幅有限，也可以只生成“统一字段结构表”，并在表前一段文字说明四个平台集合映射。

## 表二：统一字段结构设计

建议字段：

| 字段名 | 数据类型 | 字段含义 | 来源或生成方式 | 设计作用 |
|---|---|---|---|---|

真实字段如下：

| 字段名 | 数据类型 | 字段含义 | 来源或生成方式 | 设计作用 |
|---|---|---|---|---|
| `_id` | ObjectId | MongoDB 自动生成主键 | 数据库自动生成 | 保证文档唯一性 |
| `id` | String | 系统统一帖子 ID | 由平台原始 ID 加平台前缀生成 | 前端展示、详情查询和去重的主要标识 |
| `_key` | String | 去重辅助字段 | 由系统补齐或兼容历史数据生成 | 兼容历史索引和防重复写入 |
| `note_id` | String | 小红书笔记 ID 兼容字段 | 小红书平台原始笔记 ID 或系统补齐 | 兼容小红书旧字段和历史索引 |
| `platform` | String | 平台标识 | 爬虫写入，如 `weibo`、`douyin`、`tieba`、`xhs` | 支持按平台过滤和集合校验 |
| `platform_name` | String | 平台中文名称 | 后端或爬虫映射生成 | 前端展示平台来源 |
| `screen_name` | String | 作者昵称 | 平台页面或接口解析获得 | 展示发布主体 |
| `text` | String | 舆情正文内容 | 爬虫解析正文、标题或描述后生成 | 文本风险识别和前端展示 |
| `created_at` | String/Date | 平台发布时间 | 平台页面或接口解析获得 | 支持时间筛选和排序 |
| `source` | String | 数据来源或来源链接 | 爬虫采集时记录 | 辅助追溯原始信息 |
| `keyword` | String | 采集关键词 | 当前采集任务使用的关键词 | 标识数据发现来源 |
| `matched_keywords` | Array | 命中的风险关键词列表 | 正文筛选时生成 | 支持关键词解释和高级检索 |
| `topics` | Array | 话题、标签或吧名等扩展信息 | 平台解析获得 | 补充语义和传播场景信息 |
| `pics` | Array | 图片本地路径或图片链接列表 | 图片下载或链接解析生成 | 供图片检测、OCR 和前端预览使用 |
| `crawl_time` | String/Date | 爬虫采集时间 | 系统写入时间 | 支持采集记录追踪 |
| `inserted_at` | String/Date | 首次入库时间 | 首次写入 MongoDB 时生成 | 区分采集时间和入库时间 |
| `analysis` | Object | 多模态分析结果 | 后端分析模块或 API 分析后写回 | 存储文本、图片、OCR、融合评分等结果 |
| `analysis_status` | String | 分析状态 | 后端分析流程更新 | 标识 `done`、`analyzing`、`error`、`no_image` 等状态 |
| `analysis_error` | String | 分析错误信息 | 分析失败时写入 | 便于运行日志和异常排查 |
| `analysis_image` | String | 实际用于分析的图片路径 | 分析模块选择可用图片后写入 | 记录多模态分析所用图片 |
| `analysis_at` | String/Date | 分析完成或更新时间 | 分析流程写入 | 支持分析任务追踪和排序 |

## 去重与索引设计说明

表格下方可以补充以下说明：

系统采用“平台分集合存储 + 统一字段结构”的 MongoDB 设计方式。不同平台的数据分别存入对应集合，避免不同平台原始字段差异造成数据混乱；同时通过统一的 `id`、`platform`、`text`、`pics`、`analysis` 等字段保证前端展示和分析模块能够复用同一套逻辑。系统在写入时结合平台原始 ID、统一 `id`、`_key`、`note_id` 和本地 `seen_ids.txt` 进行去重，避免重复采集同一帖子。后端还提供索引清理与字段补齐逻辑，用于处理历史索引冲突和字段缺失问题。
