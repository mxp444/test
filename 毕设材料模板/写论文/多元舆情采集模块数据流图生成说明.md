# 多元舆情采集模块数据流图生成说明

## 使用位置

该图用于毕业论文第 **3.3.1 多元舆情采集模块设计** 小节，图名建议为：

**图3-X 多元舆情采集模块数据流图**

该图不是普通流程图，而是数据流图，重点表现“数据从哪里来、经过哪些处理、存到哪里、输出给谁”。

## 图形类型要求

生成一张黑白色调的数据流图，适合放入本科毕业论文。

要求：

- 使用黑白灰色调，不使用彩色。
- 字体使用宋体。
- 字号 14px。
- 画布宽度 590px。
- 图形风格必须接近传统数据流图，不要画成普通业务流程图。
- 不要使用装饰性图标。
- 箭头上标注数据流名称。
- 图中应区分外部实体、处理过程、数据存储。
- 处理过程必须编号，例如 1.1、1.2、2.1、2.2。
- 处理过程尽量使用圆形或椭圆形，不要使用普通矩形。

## 数据流图符号建议

必须采用类似截图中的传统数据流图表示法：

- 外部实体：矩形框。
- 处理过程：圆形或椭圆形，内部写编号和处理名称，例如“1.1 检查配置”。
- 数据存储：数据存储符号，使用左侧编号，例如“D1 平台配置”“D2 去重记录”。
- 数据流：带箭头连线，并在箭头旁标注数据名称。

如果生成 draw.io XML，应保证 XML 可直接打开，不要使用复杂样式。

## 必须采用的图层级

本图建议画成 **0层细化后的 1层数据流图**，即不要只画一个大过程，而要画出采集模块内部的多个处理过程。

处理过程必须按以下编号：

| 编号 | 处理过程名称 |
|---|---|
| 1.1 | 读取采集配置 |
| 1.2 | 创建采集任务 |
| 2.1 | 调度平台爬虫 |
| 2.2 | 平台数据采集 |
| 3.1 | 数据解析与字段统一 |
| 3.2 | 图片下载与路径保存 |
| 4.1 | 关键词过滤与去重 |
| 4.2 | 写入平台集合 |

数据存储必须按以下编号：

| 编号 | 数据存储名称 |
|---|---|
| D1 | 平台配置文件 |
| D2 | 去重记录 |
| D3 | 本地图片目录 |
| D4 | MongoDB平台集合 |
| D5 | 采集运行日志 |

外部实体必须包括：

| 外部实体 | 说明 |
|---|---|
| 管理员 | 提交采集平台、关键词、Cookie、页数、图片规则 |
| 互联网平台 | 微博、抖音、百度贴吧、小红书 |
| 后续分析模块 | 接收 MongoDB 中待分析的舆情数据，可画在最右侧 |

## 系统背景

本系统为“网络金融舆情风险监测系统”。系统采集微博、抖音、百度贴吧、小红书四个平台中的金融风险相关图文舆情数据，并将数据统一写入 MongoDB。采集模块由前端采集任务管理页面、Flask 后端、统一爬虫运行器和四个平台爬虫组成。

项目中的关键代码位置：

- 后端主程序：`整合/backend/app.py`
- 多平台爬虫运行器：`整合/crawlers/runner.py`
- 微博爬虫：`整合/crawlers/weibo/`
- 抖音爬虫：`整合/crawlers/douyin/`
- 百度贴吧爬虫：`整合/crawlers/tieba/`
- 小红书爬虫：`整合/crawlers/xhs/`
- MongoDB 数据库：`test`
- 平台集合：
  - 微博：`test_weibo`
  - 抖音：`test_douyin`
  - 百度贴吧：`test_baidu`
  - 小红书：`test_xhs`

## 图中必须包含的外部实体

1. **管理员**
   - 管理员在后台采集任务管理页面选择平台、配置采集规则、启动采集任务。

2. **互联网平台**
   - 包括微博、抖音、百度贴吧、小红书。
   - 可以画成一个总的外部实体“互联网平台”，内部注明四个平台名称。
   - 也可以画成四个并列外部实体。

## 图中必须包含的处理过程

必须包含以下处理过程节点，并使用圆形或椭圆形：

1. **1.1 读取采集配置**
   - 接收管理员设置的平台、关键词、Cookie、页数、图片规则、采集目标量等。
   - 从 D1 平台配置文件读取默认配置。

2. **1.2 创建采集任务**
   - 对应后端 `CrawlTask`。
   - 根据采集配置创建任务对象。

3. **2.1 调度平台爬虫**
   - 对应 `crawlers/runner.py`。
   - 根据平台 ID 动态加载微博、抖音、百度贴吧、小红书爬虫入口。
   - 注入平台运行配置。

4. **2.2 平台数据采集**
   - 从微博、抖音、百度贴吧、小红书抓取帖子或图文内容。
   - 获取正文、作者、发布时间、图片地址、关键词等原始数据。

5. **3.1 数据解析与字段统一**
   - 将不同平台原始数据整理成统一字段结构。
   - 统一字段包括：`id`、`platform`、`platform_name`、`screen_name`、`text`、`created_at`、`pics`、`matched_keywords`、`analysis_status` 等。

6. **3.2 图片下载与路径保存**
   - 下载平台图片到本地目录。
   - MongoDB 中保存图片路径和访问地址，不直接保存图片二进制。

7. **4.1 关键词过滤与去重**
   - 使用关键词过滤无关内容。
   - 使用 `seen_ids.txt`、`id`、`_key`、`note_id` 等字段避免重复写入。
   - 异常平台或异常页面应跳过，不中断整个采集任务。

8. **4.2 写入平台集合**
   - 将处理后的舆情数据写入 MongoDB 对应平台集合。

## 图中必须包含的数据存储

1. **D1 平台配置文件**
   - 包括 `runtime_config.json`、`platform_settings.json`。
   - 存储 Cookie、关键词、采集页数、图片规则、采集目标量等配置。

2. **D3 本地图片目录**
   - 保存爬取下载的舆情图片。

3. **D2 去重记录**
   - `seen_ids.txt`
   - 用于保存已采集 ID。

4. **D4 MongoDB平台集合**
   - 数据库：`test`
   - 集合：`test_weibo`、`test_douyin`、`test_baidu`、`test_xhs`

5. **D5 采集运行日志**
   - 保存或展示采集过程中产生的写入数、重复数、丢弃数、异常信息。

## 推荐数据流

请按以下数据流组织图：

1. 管理员 → 1.1 读取采集配置
   - 数据流名称：采集平台、关键词、Cookie、页数、图片规则

2. D1 平台配置文件 → 1.1 读取采集配置
   - 数据流名称：运行配置、平台配置

3. 1.1 读取采集配置 → 1.2 创建采集任务
   - 数据流名称：采集任务参数

4. 1.2 创建采集任务 → 2.1 调度平台爬虫
   - 数据流名称：平台列表、目标数量、控制状态

5. 2.1 调度平台爬虫 → 2.2 平台数据采集
   - 数据流名称：平台入口、配置覆盖项

6. 互联网平台 → 2.2 平台数据采集
   - 数据流名称：网页内容、搜索结果、帖子数据、图片链接

7. 2.2 平台数据采集 → 3.1 数据解析与字段统一
   - 数据流名称：原始舆情数据

8. 3.1 数据解析与字段统一 → 3.2 图片下载与路径保存
   - 数据流名称：图片 URL、帖子图片信息

9. 3.2 图片下载与路径保存 → D3 本地图片目录
   - 数据流名称：图片文件

10. 3.1 数据解析与字段统一 → 4.1 关键词过滤与去重
    - 数据流名称：统一字段数据

11. D2 去重记录 → 4.1 关键词过滤与去重
    - 数据流名称：已采集 ID

12. 4.1 关键词过滤与去重 → D2 去重记录
    - 数据流名称：新增采集 ID

13. 4.1 关键词过滤与去重 → 4.2 写入平台集合
    - 数据流名称：有效非重复舆情数据

14. 4.2 写入平台集合 → D4 MongoDB平台集合
    - 数据流名称：分平台结构化文档

15. 4.2 写入平台集合 → D5 采集运行日志
    - 数据流名称：写入数、重复数、丢弃数、运行日志

16. D4 MongoDB平台集合 → 后续分析模块
    - 数据流名称：待分析舆情数据
