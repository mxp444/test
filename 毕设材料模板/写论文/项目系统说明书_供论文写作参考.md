## 1. 项目总体目标

本系统面向网络金融舆情风险治理场景，目标是对多个互联网平台上的金融相关图文内容进行自动化采集，并结合文本、图像、OCR 等多模态特征进行风险识别和融合评估，最终在 Web 前端中展示舆情列表、风险等级、详细分析结果、追踪记录和后台运维配置。

系统核心目标包括：

1. 支持多平台舆情采集：微博、抖音、百度贴吧、小红书。
2. 支持金融风险关键词驱动的定向爬取。
3. 支持帖子正文、图片、本地图片路径、发布时间、用户昵称、平台来源等统一结构化存储。
4. 支持多模态风险分析：文本金融属性、风险要素、情感、煽动性、二维码、图片模糊度、色彩丰富度、设计感、OCR 文本及图文融合。
5. 支持 MongoDB 按平台集合存储，并统一前端读取。
6. 支持前台业务端与后台管理端分权。
7. 支持管理员配置采集平台、采集任务、系统运行参数、账号、用户权限等。
8. 支持普通用户查看舆情、申请采集源、追踪关注帖子，但不能直接执行采集任务或修改后台配置。
9. 支持后台数据库管理功能，管理员可在 Web 端以类似 MongoDB Compass 的方式查看数据库、集合、文档、索引数量、文档数量，并完成基础查询、分页浏览、新增文档、编辑文档、删除文档、清空集合数据、创建集合、删除集合、创建数据库、删除数据库等操作。

## 4. 系统总体架构

系统采用前后端分离但同源部署的结构。Flask 后端既提供 API，也提供前端静态文件。

架构可以在论文中描述为五层：

1. 数据采集层：四个平台爬虫。
2. 数据存储层：MongoDB + 本地图片目录 + JSON 配置文件。
3. 多模态分析层：文本、图片、OCR、融合分析。
4. 服务接口层：Flask REST API。
5. 前端展示层：业务端与后台管理端。

其中后台管理端新增数据库管理子系统，位于系统运维与数据维护范围内。该子系统不直接参与舆情风险识别流程，但为管理员提供 MongoDB 数据查看、检索和维护能力，可在论文中作为“后台数据管理模块”或“数据库可视化管理模块”描述。

推荐论文中使用如下数据流描述：

```text
用户/管理员操作前端
        ↓
Flask 后端接口
        ↓
爬虫调度器 runner.py
        ↓
四个平台爬虫抓取帖子与图片
        ↓
MongoDB 按平台集合存储原始帖子
        ↓
自动或手动调用多模态分析
        ↓
分析结果写回 MongoDB 文档 analysis 字段
        ↓
前端读取分析后数据并展示
```

数据库管理子系统的数据流可单独描述为：

```text
管理员进入后台数据库管理页面
        ↓
Flask 后端校验管理员权限
        ↓
通过 PyMongo 读取 MongoDB 数据库、集合和文档信息
        ↓
后端将 ObjectId、Binary、datetime 等 BSON 类型转换为前端可序列化结构
        ↓
前端以资源管理器和文档树形式展示
        ↓
管理员执行查询、分页、新增、编辑、删除等维护操作
        ↓
后端再次校验权限和目标库名后写回 MongoDB
```

## 6. 数据采集模块设计

### 6.1 多平台统一运行器

文件：`整合/crawlers/runner.py`

该文件定义 `PLATFORMS`：

```python
PLATFORMS = {
    "weibo": {"label": "微博", "dir": ..., "entry": "scrpy.py", "collection": "test_weibo"},
    "douyin": {"label": "抖音", "dir": ..., "entry": "scrpy.py", "collection": "test_douyin"},
    "tieba": {"label": "百度贴吧", "dir": ..., "entry": "scrpy.py", "collection": "test_baidu"},
    "xhs": {"label": "小红书", "dir": ..., "entry": "main.py", "collection": "test_xhs"},
}
```

核心职责：

- 维护平台 ID、中文名称、爬虫目录、入口文件、MongoDB 集合映射。
- `normalize_platforms()`：将前端勾选的平台列表转换为有效平台列表。
- `platform_collection()`：根据平台 ID 返回对应 MongoDB 集合。
- `crawler_import_context()`：临时切换导入路径，避免不同平台爬虫的 `setting.py` 互相污染。
- `run_platform()`：动态加载平台入口文件，注入 MongoDB 配置、平台配置覆盖项，并调用平台爬虫的 `run_crawler()`。

### 6.2 采集任务调度

文件：`整合/backend/app.py`

主要类：

- `WebCrawlController`
  - 管理暂停、继续状态。
  - 为各爬虫提供 `poll()`、`wait_if_paused()` 等控制能力。

- `CrawlTask`
  - 管理当前采集任务状态。
  - 接收前端平台选择。
  - 按平台并行启动爬虫线程。
  - 记录运行日志。
  - 聚合每个平台写入、重复、未命中、丢弃等统计。

采集任务启动接口：

- `POST /api/crawl/start`

暂停接口：

- `POST /api/crawl/pause`

状态接口：

- `GET /api/crawl/status`

普通用户不能调用启动和暂停接口。

### 6.3 平台采集均衡策略

后端使用 `CRAWL_TOTAL_TARGET` 控制单轮总采集目标数。若同时选择多个平台，系统会把总目标量近似平均分配到各个平台：

```python
platform_item_limit = ceil(CRAWL_TOTAL_TARGET / selected_platform_count)
```

这样可以避免某个平台无限采集，导致数据库中某一平台数据过多。每个平台爬虫内部也实现随机交错策略：

- 微博：`RANDOMIZE_CRAWL_ORDER`
- 百度贴吧：`RANDOMIZE_FORUM_ORDER`
- 小红书：`RANDOMIZE_CRAWL_ORDER`

## 7. 统一数据结构设计

四个平台爬虫最终都尽量写成统一字段，便于前端和分析模块复用。核心字段包括：

| 字段 | 说明 |
|---|---|
| `_id` | MongoDB 自动 ID |
| `id` | 平台内唯一 ID，前端主要识别字段 |
| `_key` | 去重辅助字段 |
| `note_id` | 兼容小红书旧字段和索引 |
| `platform` | 平台 ID，如 weibo、douyin、tieba、xhs |
| `platform_name` | 平台中文名 |
| `screen_name` | 作者昵称 |
| `text` | 帖子正文 |
| `created_at` | 平台发布时间 |
| `source` | 来源 |
| `keyword` | 命中关键词 |
| `matched_keywords` | 命中关键词列表 |
| `topics` | 话题或标签 |
| `pics` | 图片本地路径列表 |
| `crawl_time` | 爬取时间 |
| `inserted_at` | 首次写入时间 |
| `analysis` | 多模态分析结果 |
| `analysis_status` | done / analyzing / error / no_image 等 |
| `analysis_error` | 分析错误信息 |
| `analysis_image` | 实际用于分析的图片路径 |
| `analysis_at` | 分析时间 |

后端通过 `serialize_doc()` 将 MongoDB 文档转换为前端可用数据，包括：

- 将 `_id` 转字符串。
- 补齐 `id`。
- 补齐 `platform_name`。
- 补齐 `screen_name`。
- 将 `pics` 转换为 `{path, url}`，其中 `url` 由 `/media/...` 提供访问。

## 8. MongoDB 设计

数据库：`test`

集合：

| 平台 | 集合名 |
|---|---|
| 微博 | `test_weibo` |
| 抖音 | `test_douyin` |
| 百度贴吧 | `test_baidu` |
| 小红书 | `test_xhs` |

设计原则：

1. 各平台数据分集合存储，便于平台隔离和维护。
2. 前端读取时通过后端统一聚合。
3. 每个平台使用统一字段结构，避免前端为每个平台写大量适配逻辑。
4. 使用 `id`、`_key`、`note_id` 等字段兼容历史去重逻辑。
5. 后端有索引清理逻辑，避免历史唯一索引冲突。

后端相关函数：

- `normalize_collection_indexes()`：删除历史不兼容索引，如 `note_id_1`、`_key_1` 或错误 sparse `id_1`。
- `backfill_missing_unique_fields()`：补齐 `_key` 和 `note_id`。
- `normalize_documents_collections()`：如果文档存错集合，会根据 `platform` 字段移动到正确集合。
- `iter_platform_collections()`：统一遍历四个平台集合。
- `load_analyzed_items()`：读取已分析、含图、含文本的数据。

### 8.1 数据库管理子系统设计

新增后台数据库管理功能后，MongoDB 不仅作为舆情数据持久化层，也在系统后台提供可视化维护入口。该功能参考 MongoDB Compass 的基本交互，但根据毕业设计系统需求进行了简化。

数据库管理页面主要面向管理员，普通用户无权访问。其设计目标包括：

1. 在系统内直接查看 MongoDB 连接下的数据库列表。
2. 以资源管理器形式展示 `localhost:27017`、数据库、集合三级结构。
3. 支持展开和收起数据库与集合层级，方便管理多个库和集合。
4. 在集合条目右侧显示该集合当前文档数量，便于快速判断数据规模。
5. 支持查看集合概览，包括集合名称、文档数量、存储大小、平均文档大小、索引数量等。
6. 支持按 JSON 条件查询文档，例如 `{ "platform": "weibo" }` 或 `{ "analysis_status": "done" }`。
7. 支持分页浏览文档，每页数量可选 10、20、50、100。
8. 支持刷新、上一页、下一页等常用浏览操作。
9. 支持新增文档、编辑文档、删除单条文档和清空集合文档。
10. 支持创建数据库、删除数据库、创建集合、删除集合。

为避免误操作，数据库管理功能包含以下限制：

- 写操作仅管理员可用。
- `admin`、`config`、`local` 等 MongoDB 系统库禁止删除或修改。
- 删除数据库、删除集合、清空集合数据前需要确认输入指定文本。
- 文档编辑时不允许直接修改 `_id` 字段。
- 后端使用 `safe_mongo_name()` 校验数据库名和集合名，避免非法名称。

后端处理 MongoDB 文档时，需要解决 BSON 类型不能直接 JSON 序列化的问题。系统通过 `bson_to_jsonable()` 将常见 BSON 类型转换为前端可展示格式，例如：

| BSON 类型 | 前端转换方式 |
|---|---|
| `ObjectId` | 转换为字符串 |
| `datetime` | 转换为 ISO 时间字符串 |
| `Binary` / `bytes` | 转换为带类型说明和十六进制预览的对象 |
| `Decimal128` | 转换为字符串 |
| `Timestamp` | 转换为普通对象 |
| `dict` / `list` | 递归转换内部字段 |

该设计使后台页面在打开含图片二进制、系统日志或特殊 BSON 字段的集合时不会报错。

## 9. 多模态分析模块设计

### 9.1 统一接口

所有分析实现文件都需要提供：

```python
class MultimodalRiskFusion:
    def analyze(self, post_text: str, image_path: str) -> dict:
        ...

    def analyze_batch(self, items) -> list[dict]:
        ...
```

后端通过 `load_analyzer_class()` 动态加载 `runtime_config.json` 中配置的分析文件。

### 9.2 输入

单条分析输入：

- `post_text`：帖子正文。
- `image_path`：本地图片路径。

后端只对有文本且有图片的数据执行多模态分析。如果没有可用图片，则标记：

```text
analysis_status = "no_image"
```

### 9.3 输出结构

后端通过 `compact_result()` 规范分析结果，核心输出包括：

```text
input
runtime
text_feature_extraction
image_feature_extraction
ocr_text_feature_extraction
multimodal_feature_fusion
fusion_network_analysis
final_multimodal_analysis
summary
```

其中 `summary` 包含：

- `total_score`
- `risk_level`
- `conclusion`
- `suggestion`
- `reasons`
- `modality_breakdown`
- `cross_modal_features`
- `network_output`

### 9.4 文本特征

文本分析指标包括：

1. 金融属性分类 `FINANCE_LABELS`
   - 输出最可能的四个类别及置信度。

2. 风险要素 `RISK_CATEGORIES`
   - 识别各类风险词汇。
   - 统计每一类词汇及出现数量。
   - 统计总风险词数。

3. 情感标签 `SENTIMENT_LABELS`
   - 输出最可能的一个情感标签。

4. 煽动性评估 `INCITEMENT`
   - 参考 `nlp煽动性评估.py`。
   - 包括三个维度分数、维度下细分检测结果、最终总分。

### 9.5 图片特征

图片分析指标包括：

1. 二维码检测：判断图片是否含二维码。
2. 模糊度检测：给出清晰度/模糊度分数。
3. 色彩丰富度：输出总分和细节指标。
4. 设计感评估：参考 `pic设计感.py` 输出。
5. OCR 文字识别：提取图片中文字及位置。

### 9.6 OCR 文本二次分析

OCR 提取出的图片文字会再走一遍文本分析逻辑，得到：

- OCR 文本金融属性。
- OCR 文本风险要素。
- OCR 文本情感。
- OCR 文本煽动性。

这样可以发现“正文看似正常，但图片中包含高风险营销话术”的情况。

### 9.7 多模态融合

融合逻辑综合：

- 原文文本风险。
- 图片视觉风险。
- OCR 文本风险。
- 图文语义一致性。
- 图文风险要素协同。
- 私域引流特征。
- 营销煽动性。
- 视觉金融营销强度。

最终输出综合风险分和风险等级。

风险等级可按论文需要描述为：

- 低风险
- 中风险
- 中高风险
- 高风险

## 10. API 版分析设计

文件：`整合/多模态融合分析_API.py`

特点：

- 后端直接加载该文件。
- 不再依赖项目根目录 `main.py`。
- 调用阿里云 DashScope 兼容接口。
- 默认模型可由环境变量配置，默认类似 `qwen3.5-flash`。
- Prompt 中只给分类体系和输出格式，不直接给固定词库，让 API 自行抽取每类风险词。
- 输出格式与本地版保持一致，方便前端统一展示。

API 版适合：

- 快速得到完整结构化分析结果。
- 降低本地模型依赖。
- 论文中作为“基于大模型接口的多模态分析实现方案”描述。

## 11. 本地版分析设计

文件：`整合/多模态融合分析.py`

特点：

- 尝试加载本地 NLP 和图像模块。
- 若模型或依赖不可用，可以降级为启发式分析。
- 支持 `load_word_vector_model` 控制是否加载大词向量文件 `sgns.financial.word`。
- 支持 `strict_init` 控制严格启动还是降级运行。
- 输出结构与 API 版一致。

本地版适合论文中说明：

- 系统可在不调用外部大模型的情况下完成基础风险识别。
- 通过组件降级提高可用性。
- 多模态融合不是单一模型判断，而是多特征、多维度综合评分。

## 12. 自动分析与并行分析

系统支持两种分析触发方式：

1. 爬虫写入数据后自动分析。
2. 后台手动点击“补分析历史数据”。

配置项：

- `auto_analyze_crawled_items`
- `analysis_max_workers`

自动分析逻辑：

- 爬虫优先将帖子原始数据写入 MongoDB。
- 后端再调用多模态分析。
- 分析完成后将 `analysis`、`analysis_status`、`analysis_at` 等字段更新回 MongoDB。

并行分析逻辑：

- `analyze_unprocessed_documents(limit=20)` 会从四个平台集合中按平台均衡取未分析数据。
- 使用 `ThreadPoolExecutor` 并发执行。
- 每条数据先标记为 `analysis_status = "analyzing"`，防止重复任务抢占。
- 成功后写入 `analysis_status = "done"`。
- 失败后写入 `analysis_status = "error"` 和错误信息。

## 15. 平台设置与采集申请流程

   - 申请状态变为已退回。

## 16. 系统配置文件说明

### 16.1 runtime_config.json

路径：`整合/backend/runtime_config.json`

字段：

```json
{
  "analysis_engine_file": "多模态融合分析_API.py",
  "analysis_max_workers": 20,
  "auto_analyze_crawled_items": true,
  "load_word_vector_model": true,
  "strict_runtime": false,
  "crawl_total_target": 200,
  "platform_cookies": {
    "weibo": "",
    "douyin": "",
    "tieba": "",
    "xhs": ""
  }
}
```

含义：

- `analysis_engine_file`：当前使用的多模态分析实现。
- `analysis_max_workers`：并行分析线程数。
- `auto_analyze_crawled_items`：爬取后是否自动分析。
- `load_word_vector_model`：是否加载大词向量模型。
- `strict_runtime`：严格模式，依赖异常时是否终止。
- `crawl_total_target`：单轮采集目标总数。
- `platform_cookies`：运行时 Cookie 覆盖。

## 17. 风险等级与筛选逻辑

前端通过 `riskScore(item)` 从分析结果中读取综合风险分。常用风险等级：

- 高风险：`score >= 80`
- 中高风险：`60 <= score < 80`
- 中风险：`40 <= score < 60`
- 低风险：`score < 40`

风险分析页可以按全部、中风险、中高风险、高风险切换。

前端监测页也支持快速筛选：

- 全部。
- 微博。
- 抖音。
- 贴吧。
- 小红书。
- 中高风险。
- 已分析。
- 近两个月。

## 18. 关键业务流程

### 18.1 管理员启动多平台采集

1. 管理员登录。
2. 进入后台“采集任务管理”。
3. 勾选采集平台。
4. 点击启动采集。
5. 后端创建 `CrawlTask`。
6. 根据选择的平台启动对应爬虫。
7. 每个平台写入对应 MongoDB 集合。
8. 若开启自动分析，写入后自动分析。
9. 前端轮询 `/api/crawl/status` 展示实时日志和统计。

### 18.2 用户查看舆情

1. 用户登录。
2. 前端调用 `/api/items?limit=2000`。
3. 后端从四个平台集合读取已分析且有图的文档。
4. 后端序列化文档并返回。
5. 前端按平台、关键词、风险等级等条件筛选。
6. 用户可查看详情或加入追踪。

### 18.3 历史数据补分析

1. 管理员点击“补分析历史数据”。
2. 后端调用 `/api/analyze/unprocessed?limit=20`。
3. 从四个平台集合中均衡取未分析数据。
4. 并行执行多模态分析。
5. 分析结果写回 MongoDB。

### 18.4 查看详情报告

1. 用户点击帖子详情。
2. 前端进入 `/report?id=...`。
3. 后端通过 `/api/items/<doc_id>` 读取单条数据。
4. 报告页展示帖子原文、图片、综合风险、文本特征、图像特征、OCR、融合特征和建议。

### 18.5 管理员进行数据库查询与维护

1. 管理员登录系统。
2. 进入后台“数据库管理”页面。
3. 前端调用 `/api/admin/database/overview` 获取 MongoDB 数据库与集合树。
4. 管理员在左侧资源管理器中选择数据库或集合。
5. 若选择数据库，右侧显示集合概览表。
6. 若选择集合，右侧显示 Documents 文档视图。
7. 管理员可在 Filter 输入框中填写 JSON 查询条件并点击 Find。
8. 后端调用 PyMongo 查询集合，并将 BSON 类型转换为 JSON 可展示格式。
9. 前端以树形字段结构展示文档，字段对象可展开或隐藏。
10. 管理员可执行 Add Data、新增集合、删除集合、删除单条文档、清空集合数据等操作。
11. 所有写操作均通过管理员权限校验，并对系统库和危险操作进行保护。
