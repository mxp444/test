# 数据库 ER 图生成说明

## 使用位置

该图用于毕业论文 **3.3.2 MongoDB数据库设计** 小节。

图名建议：

**图3-X 系统数据库 ER 图**

该图要画成传统 ER 图，也就是教材中常见的 **实体-联系图**，风格参考“学生—注册—课程”示例图。

## 图形风格要求

必须使用传统 ER 图画法，不要使用 Crow's Foot 表格图，不要画成 UML 类图，不要画成数据流图。

画法要求：

- 实体：矩形。
- 属性：椭圆。
- 关系：菱形。
- 实体与属性之间用直线连接。
- 实体与关系之间用直线连接。
- 主键属性必须加下划线。
- 多值属性可以使用双椭圆，如果画图工具不方便，可以用普通椭圆并注明“多值”。
- 复合属性可以拆成多个椭圆，但本图为了简洁不强制拆分。
- 关系两侧标注基数，例如 `1`、`N`、`M`。
- 使用黑白灰色调。
- 字体使用宋体。
- 字号 14px。
- 画布宽度 590px。
- 图中不要使用彩色、渐变、阴影、装饰图标。

## 系统数据库背景

系统名称：网络金融舆情风险监测系统。

系统底层使用 MongoDB 和部分 JSON 文件保存数据，但在论文 ER 图中需要抽象为概念实体。

主要数据来源：

- MongoDB 数据库：`test`
- 平台舆情集合：
  - `test_weibo`
  - `test_douyin`
  - `test_baidu`
  - `test_xhs`
- 本地配置与业务文件：
  - `users.json`
  - `watchlist.json`
  - `collection_sources.json`
  - `platform_settings.json`
  - `runtime_config.json`
  - `user_activity.json`

## 建议实体

为了图面清晰，传统 ER 图不要放太多实体。建议保留以下 7 个核心实体：

1. 用户
2. 平台
3. 采集配置
4. 采集源申请
5. 舆情数据
6. 分析结果
7. 追踪记录

如果空间允许，可再加入“舆情图片”和“操作记录”两个实体；如果空间不足，可以省略它们，把图片作为“舆情数据”的属性。

## 实体与属性设计

### 1. 用户

实体名称：**用户**

属性：

- 用户名（主键，下划线）
- 密码
- 显示名称
- 用户角色
- 联系电话
- 所属部门
- 启用状态

说明：

用户对应 `users.json` 中的账号信息。用户可以是普通用户或管理员。

### 2. 平台

实体名称：**平台**

属性：

- 平台编号（主键，下划线）
- 平台名称
- 集合名称
- 爬虫目录
- 入口文件
- 启用状态

说明：

平台表示微博、抖音、百度贴吧、小红书四类采集平台。

### 3. 采集配置

实体名称：**采集配置**

属性：

- 配置编号（主键，下划线）
- Cookie
- 关键词
- 最近天数
- 最大页数
- 是否要求图片
- 更新时间

说明：

采集配置对应 `platform_settings.json`，用于保存平台 Cookie、关键词、页数、时间范围等运行参数。论文中不要展示真实 Cookie 内容。

### 4. 采集源申请

实体名称：**采集源申请**

属性：

- 申请编号（主键，下划线）
- 平台类型
- 平台名称
- 目标地址
- 申请关键词
- 审核状态
- 提交时间
- 审核时间

说明：

采集源申请对应 `collection_sources.json`，由普通用户提交，管理员审核。

### 5. 舆情数据

实体名称：**舆情数据**

属性：

- 舆情编号（主键，下划线）
- Mongo编号
- 作者昵称
- 正文内容
- 发布时间
- 命中关键词
- 话题标签
- 图片路径
- 爬取时间
- 分析状态

说明：

舆情数据是系统核心实体，对应 MongoDB 中四个平台集合的统一抽象：

- `test_weibo`
- `test_douyin`
- `test_baidu`
- `test_xhs`

虽然物理上按平台分集合存储，但 ER 图中可以抽象为一个“舆情数据”实体，并通过“平台—产生—舆情数据”关系区分来源。

### 6. 分析结果

实体名称：**分析结果**

属性：

- 分析编号（主键，下划线）
- 综合风险分
- 风险等级
- 情感倾向
- 风险类别
- 文本特征
- 图像特征
- OCR特征
- 处置建议
- 分析时间

说明：

分析结果对应舆情文档中的 `analysis`、`analysis_status`、`analysis_at` 等字段。

### 7. 追踪记录

实体名称：**追踪记录**

属性：

- 追踪编号（主键，下划线）
- 加入时间

说明：

追踪记录对应 `watchlist.json`，用于保存用户加入个人追踪的帖子。

### 8. 舆情图片（可选）

实体名称：**舆情图片**

属性：

- 图片编号（主键，下划线）
- 图片路径
- 图片访问地址
- 是否分析图片

说明：

如果想更完整，可以把图片从“舆情数据”中拆出来。若图面太复杂，可以不画该实体，只保留“图片路径”作为舆情数据属性。

### 9. 操作记录（可选）

实体名称：**操作记录**

属性：

- 记录编号（主键，下划线）
- 操作类型
- 操作详情
- 操作时间

说明：

操作记录对应 `user_activity.json`，用于记录登录、修改配置、数据库管理等后台操作。

## 关系设计

请用菱形表示关系，并在菱形中写关系名称。

### 1. 用户—提交—采集源申请

关系名称：**提交**

基数：

```text
用户 1 —— N 采集源申请
```

含义：

一个用户可以提交多个采集源申请，一条采集源申请只属于一个用户。

### 2. 用户—追踪—舆情数据

关系名称：**追踪**

这是一个多对多关系，建议通过“追踪记录”实体拆分。

基数：

```text
用户 1 —— N 追踪记录
追踪记录 N —— 1 舆情数据
```

或在图中表现为：

```text
用户 —— 追踪 —— 舆情数据
```

关系两侧标注：

```text
用户 M —— N 舆情数据
```

如果使用“追踪记录”实体，则不要再直接画 M:N。

### 3. 平台—产生—舆情数据

关系名称：**产生**

基数：

```text
平台 1 —— N 舆情数据
```

含义：

一个平台可以产生多条舆情数据，每条舆情数据只来源于一个平台。

### 4. 平台—拥有—采集配置

关系名称：**拥有**

基数：

```text
平台 1 —— 1 采集配置
```

或：

```text
平台 1 —— N 采集配置
```

如果表达当前配置，建议使用 `1:1`；如果表达历史配置，使用 `1:N`。本系统建议画为 `1:1`。

### 5. 采集配置—控制—舆情数据

关系名称：**控制采集**

基数：

```text
采集配置 1 —— N 舆情数据
```

含义：

一套采集配置可以控制产生多条舆情数据。

### 6. 舆情数据—生成—分析结果

关系名称：**生成**

基数：

```text
舆情数据 1 —— 1 分析结果
```

也可理解为：

```text
舆情数据 1 —— 0或1 分析结果
```

因为有些舆情可能尚未分析。传统 ER 图中如果不方便表达 0或1，可以标为 `1:1`，并在论文文字中说明“未分析数据暂时没有分析结果”。

### 7. 舆情数据—包含—舆情图片（可选）

关系名称：**包含**

基数：

```text
舆情数据 1 —— N 舆情图片
```

含义：

一条舆情可以包含多张图片。

### 8. 用户—产生—操作记录（可选）

关系名称：**产生**

基数：

```text
用户 1 —— N 操作记录
```

含义：

一个用户可以产生多条操作记录。

## 推荐布局

传统 ER 图建议中心布局：

中心：

- 舆情数据

左侧：

- 平台
- 采集配置

上方：

- 用户
- 采集源申请

右侧：

- 分析结果
- 舆情图片（可选）

下方：

- 追踪记录
- 操作记录（可选）

布局应尽量清楚表达：

```text
平台 —— 产生 —— 舆情数据 —— 生成 —— 分析结果
用户 —— 提交 —— 采集源申请
用户 —— 追踪 —— 舆情数据
平台 —— 拥有 —— 采集配置 —— 控制采集 —— 舆情数据
```

## 画图时必须注意

- 主键属性必须加下划线，例如“用户名”“平台编号”“舆情编号”。
- 每个实体至少连接 3 到 5 个主要属性，不要把所有字段都堆上去导致图太乱。
- 关系必须使用菱形，不要直接用一条线写文字。
- 关系两端标注 `1`、`N` 或 `M`。
- 不要把 MongoDB 物理集合全部画成独立实体，否则图会重复。
- 不要出现真实 Cookie、密码、API Key。
- 不要画接口、页面、按钮、流程步骤。

## 精简版实体建议

如果图太拥挤，最终只画以下 7 个实体：

1. 用户
2. 平台
3. 采集配置
4. 采集源申请
5. 舆情数据
6. 分析结果
7. 追踪记录

省略：

- 舆情图片
- 操作记录

此时把“图片路径”作为“舆情数据”的属性即可。

## 可直接给 GPT 的生成提示词

请生成一张传统 ER 图，不要生成 Crow's Foot 表格图、UML 类图、流程图或数据流图。图形风格参考“学生—注册—课程”的教材式 ER 图：实体用矩形，属性用椭圆，关系用菱形，主键属性加下划线，关系两端标注 1、N 或 M。使用黑白灰色调，宋体，14px，画布宽 590px。

系统为网络金融舆情风险监测系统。底层使用 MongoDB 和 JSON 文件，但 ER 图按概念模型抽象。核心实体包括：用户、平台、采集配置、采集源申请、舆情数据、分析结果、追踪记录。可选实体包括：舆情图片、操作记录。

实体属性如下：用户包含用户名（主键）、密码、显示名称、用户角色、联系电话、所属部门、启用状态；平台包含平台编号（主键）、平台名称、集合名称、爬虫目录、入口文件、启用状态；采集配置包含配置编号（主键）、Cookie、关键词、最近天数、最大页数、是否要求图片、更新时间；采集源申请包含申请编号（主键）、平台类型、平台名称、目标地址、申请关键词、审核状态、提交时间、审核时间；舆情数据包含舆情编号（主键）、Mongo编号、作者昵称、正文内容、发布时间、命中关键词、话题标签、图片路径、爬取时间、分析状态；分析结果包含分析编号（主键）、综合风险分、风险等级、情感倾向、风险类别、文本特征、图像特征、OCR特征、处置建议、分析时间；追踪记录包含追踪编号（主键）、加入时间。

关系如下：用户通过“提交”关系连接采集源申请，基数为 1:N；平台通过“产生”关系连接舆情数据，基数为 1:N；平台通过“拥有”关系连接采集配置，基数为 1:1；采集配置通过“控制采集”关系连接舆情数据，基数为 1:N；舆情数据通过“生成”关系连接分析结果，基数为 1:1；用户通过“追踪”关系连接舆情数据，基数为 M:N，建议用追踪记录实体拆分为用户 1:N 追踪记录、追踪记录 N:1 舆情数据。图中不要展示真实 Cookie、密码或 API Key。
